# recruiting-reminder

把邮箱里的校招笔试/测评和面试邮件,整理成飞书多维表格里的待办记录,并自动在飞书日历里安排日程。同步笔试(公司定时开考)和面试按邮件给的固定时间段建日程,异步笔试(只给截止)在截止前 24 小时内找空档做。一次跑完一次汇报,不做后台轮询。

## OfferLoop 新用户入口

用户第一次使用、没有配置 IMAP、尚未选择 lark-cli profile，或不确定缺少什么权限时，先调用同仓库的 `offerloop-setup`。只配置本 Skill 所需的邮箱、Base 和个人日历能力，不要求用户先启用 `job-collection`。

本 Skill 的日历查询和创建必须使用选定 profile 的 user 身份；不得用 bot 身份查询用户主日历。若同一 profile 已被 `job-collection` 用作 bot，仍需单独验证 user 授权，但不需要再创建一个飞书应用。

## 何时触发

用户说出下列任意意图时调用本 skill:

- "查笔试面试邮件""看一下邮箱里有没有笔试面试""安排面试""把校招邮件排进日历""跑一下 recruiting-reminder"
- 用户提到校招笔试、在线测评、面试通知、HackerRank/牛客/LeetCode 评测、行测、性格测试、一面/二面/三面/HR面,且希望批量整理

不要在以下场景触发:

- 只想读一封特定邮件 → 直接读 IMAP
- 只想在飞书日历建一个日程,与校招无关 → 直接用 `lark-calendar`
- 只想操作多维表格,与校招无关 → 直接用 `lark-base`

## 完整流程概览

1. 读邮件(用本 skill 自带的 `scripts/fetch_mail.py`)
2. 识别校招邮件(笔试/测评 + 面试),抽取关键字段
3. **第一道确认**:把识别结果给用户看,确认后才往下走
4. 写入飞书多维表格(用 `lark-base`,第一次跑自动建表)
5. 计算日程时间:同步笔试和面试按邮件给的固定时间精准建,异步笔试在截止前 24 小时内找 2 小时空档(找不到扩到 48 小时)
6. **第二道确认**:把提议日程给用户看,确认后才建
7. 建日程(用 `lark-calendar`):默认 5 分钟提醒(飞书 `+create` shortcut 默认值);如需自定义提醒时间,在日历客户端手改
8. 已处理邮件 ID 写本地 JSON,下次跳过

两道确认都要做。第一道防抽取错误(公司名/时间错了后面全错),第二道防时间安排不合适(找的空档你可能有别的计划)。

## 步骤详解

### Step 1. 读邮件

调用本 skill 自带的 `scripts/fetch_mail.py` 读 IMAP 邮箱最近 7 天邮件(纯 IMAP,无外部依赖)。默认 7 天,用户说"最近 14 天""上个月"等就按用户说的。

```bash
python3 scripts/fetch_mail.py --days 7 --max 50 --with-body
```

> 路径相对于 skill 根目录。skill 自身可能在 `~/.claude/skills/` 或项目内（如 `OfferLoop/skills/`）。从 skill 根目录调用就用 `scripts/fetch_mail.py`，从别的目录调用就用绝对路径或先 `cd` 到 skill 根目录。Python 脚本内部不依赖调用路径。

**首次使用前置**:默认从 `~/.config/offerloop/recruiting-reminder/.env`（遵循 `XDG_CONFIG_HOME`）读取 IMAP 配置；也可通过 `OFFERLOOP_IMAP_ENV` 或 `--env-file` 指定。缺失时先调用 `offerloop-setup` 初始化模板，再让用户在本机填写 IMAP_HOST / IMAP_PORT / IMAP_LOGIN / IMAP_PASSWORD / TZ。旧版 `scripts/.env` 只作兼容迁移。

对网易 163/126/yeah 邮箱,`fetch_mail.py` 会自动发 IMAP `ID` 命令绕过风控(其它邮箱不发)。这是按 host 域名自动判断的,用户不需要管。

输出 JSON,每封邮件含 `uid` / `subject` / `from` / `date` / `body_preview`(已抽取正文文字,HTML 邮件会自动去标签)。

先看 subject + from 跳过明显无关的(广告、订阅、个人邮件、安全提醒),对剩下的候选邮件逐封读完整正文:

```bash
python3 scripts/fetch_mail.py --body <uid>
```

跳过已经在 `processed_emails.json` 里的 uid(见 Step 8)。

### Step 2. 识别校招邮件并抽取字段

**识别规则**:LLM 判断 + 关键词兜底。主题或正文满足任一即视为校招相关:

- 笔试关键词:笔试、assessment、online test、coding test、HackerRank、牛客、LeetCode、测评、行测、性格测试、talent assessment、SHL、北森、智鼎
- 面试关键词:面试、一面/二面/三面/HR面、技术面、业务面、群面、视频面试、现场面、interview、video interview
- LLM 判断:即使没有关键词,如果邮件内容明显是校招笔试/测评/面试邀请,也视为校招邮件

**类型判定**:有面试关键词或轮次含"面" → 面试;否则 → 笔试。

**笔试子类型判定(重要)**:笔试分两种,直接影响后续日程逻辑——
- **同步笔试**:邮件里给了明确的开始时间(以及通常的结束时间/时长),例如"笔试时间:7月9日 14:00 开始,时长 90 分钟""2026年7月9日 14:00-15:30 在线笔试"。判定信号:正文里出现"笔试时间:X月X日 HH:mm 开始""同步笔试""统一笔试""实时笔试"等明确起点。
- **异步笔试**:邮件只给截止时间,不指定开始时间,例如"请在 7月10日 23:59 前完成""截止时间:7月10日 15:30"。判定信号:只有"截止"" deadline ""请在 X 前完成",无明确起点。

为什么要分:同步笔试公司定的固定时间,你冲突也得调开,跟面试一样精准建;异步笔试只给 deadline,需要在截止前自己找空档做。两类混在一起处理会出错。

**抽取字段**(从主题 + 正文 + 发件人综合判断):

| 字段 | 说明 | 笔试 | 面试 |
|------|------|------|------|
| 公司 | 招聘公司名,从发件人署名或正文提取 | 必填 | 必填 |
| 业务线 | 公司内的业务线/部门(如字节-抖音、腾讯-微信)。邮件没明确指出可留空 | 不适用 | 选填 |
| 岗位 | 应聘岗位,可能多岗位,用顿号分隔 | 不适用 | 必填 |
| 类型 | 笔试 / 面试 | 必填 | 必填 |
| 笔试子类型 | 同步笔试 / 异步笔试。判定信号见 Step 2 「笔试子类型判定」。**兜底:邮件没有明确开始时间时一律视为异步笔试**(即使注了"同步笔试"字样或给了时长,也按异步走找空档逻辑,避免把没有固定起点的时间当精准时间建) | 选填但强烈建议填 | 不适用 |
| 轮次 | 笔试:笔试/性格测评/行测/技术笔试;面试:一面/二面/三面/HR面/技术面/业务面/群面 | 必填 | 必填 |
| 企业性质 | 互联网/金融银行/外企/央国企/其他私企,按下方判定规则 | 必填 | 必填 |
| 开始时间 | 同步笔试的固定开始时刻、面试的固定开始时刻(ISO 8601) | 同步笔试必填 / 异步笔试不要求 | 必填 |
| 结束时间 | 同步笔试的固定结束时刻、面试的固定结束时刻(ISO 8601) | 同步笔试必填 / 异步笔试不要求(或用开始+时长推算) | 必填 |
| 截止时间 | 异步笔试的最后提交时刻(ISO 8601) | 异步笔试必填 / 同步笔试不要求(同步笔试的截止通常就是结束时间) | 不要求 |
| 时长 | 限时多少分钟 | 否(同步笔试通常可从开始-结束推算) | 通常可从开始-结束推算 |
| 平台 | 笔试:HackerRank/牛客/北森/SHL 等;面试:腾讯会议/飞书视频/Zoom/线下面试地点 | 否 | 否 |
| 链接 | 测评入口 或 视频会议 URL | 否 | 否 |
| 注意事项 | 笔试:摄像头/双机位/不能切屏;面试: dress code/携带材料/到场时间 | 否 | 否 |
| 邮件 ID | fetch_mail.py 返回的 `uid` | 必填 | 必填 |

**企业性质判定**:按 job-collection 的优先级规则 `互联网 > 金融银行 > 外企 > 央国企 > 其他私企`,命中即归,从公司名 + 发件人署名 + 邮件正文综合判断。常见判定信号:

- 互联网:字节、腾讯、阿里、百度、美团、京东、拼多多、网易、快手、小红书、滴滴、携程、B站、米哈游等
- 金融银行:四大行/股份制银行/城商行、券商、基金、保险、信托(中金、中信、招商银行、工商银行、华泰证券等)
- 外企:微软、Google、Apple、Amazon、宝洁、联合利华、麦肯锡、波士顿、德勤、普华永道等
- 央国企:中国移动、国家电网、中石油、中石化、中国建筑、招商局、中粮、五矿等
- 其他私企:上述四类都不命中的私企

判定不确定时默认 `其他私企`,不要硬塞。

字段缺失处理:
- **异步笔试无截止时间** → 跟用户说"识别到一封疑似笔试但没截止时间的邮件,跳过",附上主题让用户自己判断
- **同步笔试无开始时间但有时长** → 视为异步笔试处理(走找空档逻辑)
- **面试无开始/结束时间** → 同上,跳过并告诉用户
- 面试有开始无结束(或反之) → 用开始 + 默认 60 分钟补全

时间解析注意:

- 邮件里写"7月10日 19:00"→ 补当前年份,生成 `<年>-07-10T19:00:00<tz>`,时区用 OfferLoop IMAP 配置里的 `TZ`(默认 `Asia/Shanghai`,用户可改)
- 写"7月10日前完成"→ 笔试截止时间,转成 `<年>-07-10T23:59:00<tz>`
- 写"7月10日 19:00-20:30" → 面试开始 19:00,结束 20:30
- 写"本周五前"→ 按当前日期推算
- 时区以邮件落款时间为准,落款没有时区按 `.env` 里的 `TZ`

时间转换要调用 `date` 命令或 python 脚本,不要心算。

**多轮次处理**:一公司的不同轮次(一面/二面)是不同邮件、不同 uid,自然建不同记录,不要合并。抽"轮次"字段时务必从邮件主题里取准(比如"XX公司2026校招一面通知"→"一面"),不要都填空或填"面试"。

### Step 3. 第一道确认:信息确认

把识别到的校招邮件列表用 Markdown 表格列出来,每封一行:

```
| # | 类型 | 公司 | 企业性质 | 业务线 | 岗位 | 轮次 | 开始时间 | 结束时间 | 截止时间 | 平台 | 链接 | 注意事项 |
|---|------|------|----------|--------|------|------|----------|----------|----------|------|------|----------|
| 1 | 笔试 | 字节 | 互联网 | - | - | 笔试 | - | - | 2026-07-12 23:59 | 牛客 | https://... | 双机位 |
| 2 | 面试 | 腾讯 | 互联网 | 微信 | 产品经理 | 一面 | 2026-07-10 19:00 | 2026-07-10 20:00 | - | 腾讯会议 | https://... | 正装 |
```

然后明确问用户:

> 上面是识别到的 N 封校招邮件,信息对吗?
> - 哪些要写入 Base 并安排日历?(默认全部,可以输"跳过 2""只安排 1 和 3")
> - 有字段抽错了吗?(比如公司名错了、时间错了,直接告诉我)

用户确认前不要写 Base、不要建日历。

### Step 4. 写入飞书多维表格

**表结构**:笔试和面试分开建 Base,各自一张主表 + 5 个按企业性质分的子表。两类字段差异大(笔试有子类型/截止时间、面试有轮次),混在一份 schema 里全是冗余空列,所以拆开。

| Base | 主表名 | 子表(5 张) | 用途 |
|------|--------|------------|------|
| 校招笔试 Base | 校招笔试 | 互联网 / 金融银行 / 外企 / 央国企 / 其他私企 | 所有笔试/测评记录 |
| 校招面试 Base | 校招面试 | 同上 | 所有面试记录 |

按 Step 2 判定的类型,选对应的 Base 写入,不要写错表。

**笔试 Base schema(16 列)**:

| 字段名 | 类型 | 说明 |
|--------|------|------|
| 公司 | 文本 | |
| 笔试类型 | 单选 | 选项:笔试/性格测评/行测/技术笔试 |
| 笔试子类型 | 单选 | 选项:同步笔试/异步笔试 |
| 企业性质 | 单选 | 选项:互联网/金融银行/外企/央国企/其他私企。按 Step 2 判定规则 |
| 链接 | 超链接 | 测评入口 URL |
| 开始时间 | 日期 | 含时分。同步笔试的固定开始时间 |
| 结束时间 | 日期 | 含时分。同步笔试的固定结束时间 |
| 截止时间 | 日期 | 含时分。异步笔试的最后提交时间 |
| 平台 | 文本 | HackerRank/牛客/北森/SHL 等 |
| 注意事项 | 文本 | 摄像头/双机位/不能切屏 等 |
| 日历状态 | 单选 | 选项:未建日程/已建日程 |
| 完成状态 | 单选 | 选项:待完成/已完成/已错过 |
| 来源邮件ID | 文本 | 用于去重,fetch_mail.py 的 uid |
| 已建日程ID | 文本 | 空着,建完日程回填 |
| 关联记录ID | 文本 | 同公司不同笔试轮次的关联。首条留空;后续记录填同公司上一条记录在**本表内**的 record_id |
| 求职记录ID | 文本 | 关联 job-collection Base 主表里该公司线索的 record_id。见下方"信息源关联" |

**面试 Base schema(15 列)**:

| 字段名 | 类型 | 说明 |
|--------|------|------|
| 公司 | 文本 | |
| 业务线 | 文本 | 公司内业务线/部门,邮件未明确指出时留空。同公司不同业务线/不同岗位的面试各建一行,不合并 |
| 岗位 | 文本 | |
| 轮次 | 单选 | 选项:一面/二面/三面/HR面/技术面/业务面/群面 |
| 企业性质 | 单选 | 选项同笔试 Base |
| 链接 | 超链接 | 视频会议 URL |
| 开始时间 | 日期 | 含时分 |
| 结束时间 | 日期 | 含时分 |
| 平台 | 文本 | 腾讯会议/飞书视频/Zoom/线下地点 |
| 注意事项 | 文本 | dress code/携带材料/到场时间 |
| 日历状态 | 单选 | 选项:未建日程/已建日程 |
| 完成状态 | 单选 | 选项:待完成/已完成/已错过 |
| 来源邮件ID | 文本 | 用于去重 |
| 已建日程ID | 文本 | 空着,建完日程回填 |
| 关联记录ID | 文本 | 同公司不同轮次的关联。首条留空;后续填同公司上一轮在本表内的 record_id |
| 求职记录ID | 文本 | 可选关联 job-collection「企业清单」中的主表 record_id |

> 笔试和面试都可以指向同一条求职记录；一条申请对应多轮面试是合法的一对多关系，不要因为已经关联过一面就拒绝关联二面。

**关联记录ID 填充规则**:
- 笔试 Base 内:同一公司若有多条笔试(罕见,比如先性格测评再技术笔试),按时间顺序串。第一条留空;第二条填第一条在本表内的 record_id
- 面试 Base 内:同一公司的一面/二面/三面按时间顺序串。第一条留空;第二条填第一条在本表内的 record_id
- 主表和子表各自维护本表内的关联链,跨表不直接关联

**求职记录关联**(同包兄弟 Skill 可选联动):

笔试和面试表都保留「求职记录ID」，用于指向 `job-collection` 的「企业清单」主表 record_id。关联按以下顺序执行：

1. 优先读取 `~/.config/offerloop/config.json` 中用户确认保存的 `target_base_url`；没有时再询问用户，不按 Base 标题猜测。
2. 在「企业清单」中先按规范化公司名筛选，再使用邮件岗位与「招聘岗位」重叠、招聘批次和 `投递进度 = 已投递` 缩小候选。
3. 只有一个高置信候选时自动关联；存在多个候选时列出公司、批次、项目和岗位让用户选择；没有候选时留空，不阻断笔试或面试记录。
4. 笔试关联成功后，将目标记录「是否笔试」更新为包含「要笔试」；面试只写「求职记录ID」，不改「是否笔试」。

没安装或没初始化 `job-collection` 时，「求职记录ID」照常留空。本 Skill 必须能够独立运行。

**第一次运行**(`~/.config/offerloop/recruiting-reminder/base_config.json` 不存在或为空):

1. 先完整读取已安装的 `lark-base` Skill，再用 `+base-create` 建**两个** Base:「校招笔试」和「校招面试」。`+base-create` 接受 `--fields` 一次性建主表(默认表会被自动删,用 `--table-name` 指定的表会成为主表)。字段 JSON 必须遵循当前 `lark-base` 的字段规范，不凭记忆构造
2. 两个 Base 各自用 `+table-create` 建 5 个子表:「互联网」「金融银行」「外企」「央国企」「其他私企」,每张子表用所属 Base 的同一份 schema
3. 建完把两个 Base 的 `base_token`、主表 `table_id`、各子表 `table_id` 写入 `~/.config/offerloop/recruiting-reminder/base_config.json`，文件权限设为 `0600`。不要写入 Skill 安装目录:

```json
{
  "written_exam": {
    "base_token": "bascn...",
    "main_table_id": "tbl...",
    "sub_tables": {
      "互联网": "tbl...",
      "金融银行": "tbl...",
      "外企": "tbl...",
      "央国企": "tbl...",
      "其他私企": "tbl..."
    }
  },
  "interview": {
    "base_token": "bascn...",
    "main_table_id": "tbl...",
    "sub_tables": {
      "互联网": "tbl...",
      "金融银行": "tbl...",
      "外企": "tbl...",
      "央国企": "tbl...",
      "其他私企": "tbl..."
    }
  }
}
```

4. 之后所有写入都用这份配置,不再重复建表。

**后续运行**:

每封确认要写入的邮件,按类型选 Base,在对应 Base 内执行**双写**:

1. 按 Step 2 判定类型 → 笔试走「校招笔试」Base,面试走「校招面试」Base
2. 在该 Base 内判定企业性质,确定对应子表名
3. 写主表:用 `+record-batch-create` 把该 Base 的 schema 全列写入主表,拿到 `main_record_id`。同公司已有旧记录时,`关联记录ID` 填上一条记录在**主表内**的 record_id,否则留空
4. 写子表:用 `+record-batch-create` 把同一份列写入对应子表,拿到 `sub_record_id`。同公司已有旧记录时,`关联记录ID` 填上一条记录在**子表内**的 record_id,否则留空
5. 如果已配置 job-collection 联动，按「求职记录关联」规则消歧后，将 record_id 写到「求职记录ID」；笔试再更新对应求职记录的「是否笔试」，面试不修改该字段

写入前用 `来源邮件ID` 在对应 Base 的主表查一下是否已存在,避免同封邮件重复双写。

字段值格式遵循已安装的 `lark-base` CellValue 规范:

- 日期用 `YYYY-MM-DD HH:mm:ss` 格式(带时区的本地时间)
- 超链接用 URL 字符串

写入后 `日历状态` 默认 `未建日程`,`完成状态` 默认 `待完成`。

### Step 5. 计算日程时间

分三类,核心区别:同步笔试和面试都是公司定的固定时间,精准建;异步笔试只给截止,需要在截止前自己找空档。

**情况 A — 同步笔试(邮件给了明确开始时间,精准建)**:

跟面试逻辑一样,不找空档,不跳过用户日历冲突。

- 开始 = 邮件里的开始时间
- 结束 = 邮件里的结束时间(若只有开始,默认 +90 分钟;若有时长,用开始+时长)
- 提醒:默认 5 分钟(飞书 `+create` shortcut 默认值,下同)

> 兜底:若识别为同步但邮件里没有明确开始时间(只有时长或只有截止),降级走情况 B 异步笔试逻辑。同步笔试要求"明确起点"——没有起点就不是公司定时开考,不该精准建。

为什么精准建:同步笔试是公司统一定时开考,你错过时间就错过,不是你能调的。跟面试一个性质,日历里看到冲突自己决定要不要协调,不是 skill 该兜的。

**情况 B — 异步笔试(只有截止时间,找空档)**:

要在截止前 48 小时内找一段 2 小时空档,优先选最后 24 小时内的时段。流程:

1. 用 `lark-cli calendar +freebusy --profile <profile> --as user` 查截止前 48 小时窗口的忙闲
2. 在窗口内枚举所有 2 小时时间段,排除:
   - 23:00 ~ 08:00(夜间不排)
   - 12:00 ~ 14:00(午休不排)
   - 用户已有日程的时段(从 freebusy 结果判断,**包括本批已被同步笔试/面试占用的时段**)
3. 优先选截止前 24 小时内的最早一段合格空档;如果 24 小时内没有,再往前扩到 48 小时窗口内找
4. 取最早一段合格的作为提议
5. 如果 48 小时窗口内所有时段都被占,或距截止已不足 2 小时 → 退化为在截止前 24h 建一个 30 分钟提醒日程(只起提醒作用,不做笔试),默认 5 分钟提醒
6. 时长未明确默认 90 分钟(在 2 小时空档内做 90 分钟笔试 + 30 分钟缓冲)
7. 提醒:默认 5 分钟

**重点:同一批邮件里,先处理同步笔试和面试(它们的时间是公司定的,不可调),再处理异步笔试(找空档时把前者建好的日程当成"已占时段"排除)**。否则会出现异步笔试提议的空档跟同步笔试撞车。

**情况 C — 面试(有固定开始/结束时间,精准建)**:

- 开始 = 邮件里的开始时间
- 结束 = 邮件里的结束时间(若只有开始,默认 +60 分钟)
- 不找空档,不跳过用户日历冲突(公司定的固定时间,你看到日历后自己决定要不要调)
- 提醒:默认 5 分钟

时间计算用 python 脚本或 `date` 命令,不要心算。ISO 8601 字符串生成走统一函数。

### Step 6. 第二道确认:日程确认

把提议日程列出来:

```
| # | 公司 | 业务线 | 岗位 | 类型 | 轮次 | 子类型 | 建议时间 | 时长 | 提醒 |
|---|------|--------|------|------|------|--------|----------|------|------|
| 1 | 百度 | - | - | 笔试 | 笔试 | 同步 | 2026-07-09 14:00-15:30 | 90min | 提前5min |
| 2 | 字节 | - | - | 笔试 | 笔试 | 异步 | 2026-07-09 16:00-17:30 | 90min | 提前5min |
| 3 | 美团 | - | - | 笔试 | 笔试 | 异步 | 2026-07-12 09:00-09:30 | 30min | 截止前24h(退化提醒) |
| 4 | 腾讯 | 微信 | 产品经理 | 面试 | 一面 | - | 2026-07-10 19:00-20:00 | 60min | 提前5min |
```

明确问用户:

> 上面是建议的日程时间,行不行?
> - 异步笔试的空档时间不合适可以改(比如"字节那个改到 7/11 晚上 8 点")
> - 同步笔试和面试的时间一般不动(公司定的),如果你需要可以跳过不建
> - 不想建某条可以跳过

用户确认后再调 `lark-calendar` 建日程。

### Step 7. 建日程并回写 Base

调用 `lark-calendar +create --profile <profile> --as user` 建日程。参数:

- `summary`:
  - 笔试:`公司-笔试`(例:`字节-笔试`)
  - 面试:有业务线时 `公司-业务线-岗位-轮次`(例:`腾讯-微信-产品经理-一面`);业务线留空时退化为 `公司-岗位-轮次`(例:`腾讯-产品经理-一面`)
- `description`:包含链接、平台、注意事项的 Markdown 文本(不要塞邮件正文)。同步笔试额外注明"同步笔试(公司定时开考)";异步笔试注明"异步笔试(截止前自找空档)"
- `start_time` / `end_time`:ISO 8601
- `reminders`:默认 5 分钟(飞书 `+create` shortcut 默认值,不支持 `--reminders` 参数)

> ⚠️ 提醒时间不可在创建时自定义。`lark-cli calendar +create` shortcut 默认 `reminders: [{"minutes": 5}]` 且不接受 `--reminders` 参数。完整 API `lark-cli calendar events create` 虽然支持自定义 reminders,但对 `description` 字段在某些飞书租户下会报 `99992402 field validation failed`(原因未定位,可能是租户策略)。综合权衡后本 skill统一用 `+create` shortcut,接受 5 分钟默认提醒。如果用户需要「笔试提前 15min」「面试提前 1h」这种自定义提醒,告诉用户去飞书日历客户端手动改单条日程。

建完后拿到 `event_id`,要**双向回写**对应 Base 的主表和子表(按 Step 4 的类型分 Base:笔试回写校招笔试 Base,面试回写校招面试 Base)。Step 4 双写时存下的 `main_record_id` 和 `sub_record_id` 都要回填:

```bash
# 回写主表(从 base_config.json 按 written_exam 或 interview 取 token 和 main_table_id)
lark-cli base +record-batch-update --profile <profile> --as user \
  --base-token <token> --table-id <main_table_id> \
  --json '{"record_id_list":["<main_record_id>"],"patch":{"已建日程ID":"<event_id>","日历状态":"已建日程"}}'

# 回写子表(同 Base 的对应 sub_table_id)
lark-cli base +record-batch-update --profile <profile> --as user \
  --base-token <token> --table-id <sub_table_id> \
  --json '{"record_id_list":["<sub_record_id>"],"patch":{"已建日程ID":"<event_id>","日历状态":"已建日程"}}'
```

`+record-batch-update` 没有 `+record-update` 子命令,用 `record_id_list`(一条也走 batch)+ `patch`(同一份 patch 应用到所有目标记录)。回写单条就直接传一个 record_id。主表和子表各跑一次,保证两边的 `已建日程ID` / `日历状态` 同步。

### Step 8. 标记已处理

每封成功处理的邮件(无论是否建了日程,只要走过确认流程),把 fetch_mail.py 返回的 `uid` 追加到 `~/.local/state/offerloop/recruiting-reminder/processed_emails.json`（遵循 `XDG_STATE_HOME`）:

```json
{
  "processed_ids": ["1693642661", "1693642662", "1693642664"]
}
```

下次 Step 1 读邮件时跳过这些 uid。

**强制重新处理某封邮件**:用户说"重新检查 <uid> 号邮件"或类似时,从 `processed_emails.json` 的 `processed_ids` 数组里删掉那个 uid(用 Edit 工具直接改 JSON 文件,不要手摸命令行),然后重跑 Step 1。下一轮 fetch_mail.py 会把它当新邮件重新进入流程。

注意:重处理不会自动清理上一轮在 Base 里建的旧记录或日历里建的旧日程——如果旧记录还在,会出现两条相同来源邮件ID 的记录。重处理前先问用户:旧的 Base 记录和日程要不要删?(默认删,用 `+record-batch-delete` 按 record_id 删 Base 记录,用 calendar 删 event_id;用户说"留着"才不删)

## 本地状态文件

配置和状态必须放在用户目录，避免 Skill 更新覆盖:

- `~/.config/offerloop/recruiting-reminder/base_config.json` — 两个 Base 的配置,嵌套结构:`written_exam` 和 `interview` 各自含 `base_token`、主表 `main_table_id`、5 个子表 `sub_tables` 映射
- `~/.local/state/offerloop/recruiting-reminder/processed_emails.json` — 已处理邮件 ID 列表,去重用

两个文件都不存在时第一次运行会自动创建。

## 依赖的现有 skill

- `offerloop-setup` — 首次环境检查、profile 选择、IMAP 模板初始化
- `scripts/fetch_mail.py`(本 skill 自带)— 通用 IMAP 邮箱读取,网易系邮箱自动发 `ID` 命令绕过风控。配置默认走 OfferLoop 用户配置目录,纯 Python 标准库,无外部依赖
- `lark-base` — 建/写飞书多维表格
- `lark-calendar` — 查忙闲、建日程

调用时去读对应 skill 的 SKILL.md,不要凭记忆调 API。

## 失败恢复

- `fetch_mail.py` 报 `missing env var: IMAP_HOST`(或其它 IMAP_*):调用 `offerloop-setup` 初始化 `~/.config/offerloop/recruiting-reminder/.env`,再由用户在本机填好 IMAP_HOST/PORT/LOGIN/PASSWORD/TZ。
- `fetch_mail.py` 报 "Unsafe Login. Please contact kefu@188.com" 或登录失败:网易系邮箱需要 IMAP/SMTP 服务开启 + 授权码(不是登录密码)。去对应邮箱设置里开 IMAP/SMTP 并生成授权码,填到 `IMAP_PASSWORD`。其它邮箱报"Authentication failed"同理,确认 IMAP 已开 + 用的是 app password / 授权码而不是账户登录密码。
- Gmail:必须开 2FA + 在 https://myaccount.google.com/apppasswords 生成 app password。
- `lark-cli` 没认证:转 `offerloop-setup`，用选定 profile 的 user 身份按最小 scope 完成 split-flow 授权。除日程创建和更新外，异步笔试排期还需要 `calendar:calendar.free_busy:read`。
- `lark-base` 建表失败:把错误原样给用户,不要静默吞。两个 Base 各自独立建,一个 Base 建失败不影响另一个 Base 的流程,失败的那个 Base 重跑 Step 4 的建表流程,成功的那个不用动
- 主表或某张子表建失败:该 Base 作废,告诉用户重跑该 Base 的 Step 4 建表流程(另一个 Base 照常工作),不要在缺表状态下继续双写
- 双写时主表写成功、子表写失败:回滚主表那条记录,告诉用户该封邮件跳过,不要出现主表有记录但子表没有的脏状态。回滚命令:
  ```bash
  lark-cli base +record-batch-delete --profile <profile> --as user \
    --base-token <token> --table-id <main_table_id> \
    --record-ids '["<main_record_id>"]'
  ```
- 找空档时 `+freebusy` 失败:不直接建日程,告诉用户"无法读取你的日历忙闲,本次跳过异步笔试的自动排期,你可以手动选时间"
- 求职记录关联失败或存在多个候选:不阻断记录建立；「求职记录ID」留空或等待用户消歧，不自动更新「是否笔试」。

## 不要做的事

- 不要自动连续轮询邮箱(skill 是模型主动调用,定时轮询要靠 cron 或 hook,不归本 skill 管)
- 不要跳过任何一道确认(信息确认 + 日程确认)
- 不要把邮件正文整段塞进日历描述(描述只放链接、平台、注意事项)
- 不要在同一天里给同一封笔试邮件建多个日程
- 不要凭印象调 lark-cli 子命令,先读对应 skill 的 SKILL.md
