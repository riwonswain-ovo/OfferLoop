# 旧版双 Base 兼容

仅当 `reminder_base_url` 未配置且用户已有
`~/.config/offerloop/recruiting-reminder/base_config.json` 时读取本文件。该模式用于迁移前
继续正常使用，不用于新用户初始化。

## 定位

配置含 `written_exam` 与 `interview` 两个节点，各自保存 `base_token`、主表
`main_table_id` 和五张企业性质子表 ID。不得按名称重建资源，也不得把旧资源写入新的统一
schema。

## 旧笔试字段

公司、笔试类型、笔试子类型、企业性质、链接、开始时间、结束时间、截止时间、平台、
注意事项、日历状态、完成状态、来源邮件ID、已建日程ID、关联记录ID、求职记录ID。

## 旧面试字段

公司、业务线、岗位、轮次、企业性质、链接、开始时间、结束时间、平台、注意事项、
日历状态、完成状态、来源邮件ID、已建日程ID、关联记录ID、求职记录ID。

## 写入规则

1. 笔试只写旧笔试 Base，面试只写旧面试 Base。
2. 每个 Base 内仍按原企业性质子表双写。
3. 写入前按旧 `来源邮件ID` 查重；旧记录可能只有 IMAP UID，继续兼容该值。
4. 日历创建后回填主表和子表的日历状态与 ID。
5. 不把旧 `求职记录ID` 解释为新 `求职进展` record ID；迁移前维持原语义。
6. 不创建新旧混合字段，不推进统一求职阶段，不刷新统一工作台模块。

发现旧表缺字段、缺子表或映射损坏时只报告，不通过删除重建修复。用户确认迁移后才切换
到 `reminder_base_url`；旧 Base 永久保留为回滚入口。
