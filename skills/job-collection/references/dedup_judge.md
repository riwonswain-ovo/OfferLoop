# 跨来源招聘信息去重

本规则只处理用户登记的飞书 Base 和腾讯 Smartsheet。主表与分类子表之间的双写不是重复；去重只判断目标主表中是否已存在同一招聘条目。

## 判定顺序

1. 规范化 `投递链接` 完全一致：`high` confidence 重复。
2. 规范化 `公告链接` 完全一致：`high` confidence 为同一来源条目。
3. URL 不同：比较规范化公司、招聘批次、招聘项目和公告标题。
4. 同公司不同真实批次，例如“暑期实习”和“秋招提前批”：不是重复。
5. 公司或批次无法确认：`low` confidence，不自动合并。

URL 规范化应移除 fragment 和常见跟踪参数，保留影响资源定位的业务参数。可复用 `scripts/sync_utils.py`。

## 与目标主表比较

同步开始前读取目标主表必要字段并建立索引：

- `投递链接` → 主记录。
- `公告链接` → 主记录。
- `公司 + 招聘批次 + 招聘项目/标题` 指纹 → 候选主记录。

两个来源按同一份索引处理。每写入或补全一条记录后立即更新内存索引，避免同一轮两个来源互相重复写入。

## 重复项合并

命中已有记录时：

- 不创建新编号。
- 只补充旧记录为空、且新来源可验证的来源字段。
- `信息更新时间` 可更新为更晚的真实来源时间。
- `来源平台` 可记录多个来源名称。
- 不覆盖 `投递进度`、`编号`、主子表 record_id 和用户备注。
- 主表补全后同步补全对应子表。

## 低置信度冲突

无人值守任务中不得等待用户现场确认，也不得静默合并、删除或写入冲突候选。汇报格式：

```text
待确认重复：
1. 示例公司 - 秋招（来源 A）
   对比：示例公司 - 秋季校园招聘（来源 B）
   差异：投递链接和公告链接均不同，项目名称相似
```

用户明确判为同一条后补全旧记录；判为不同条后再新增。

## LLM 复判输出

需要语义复判时只输出：

```json
{
  "is_duplicate": true,
  "confidence": "high",
  "reason": "公司、届次、批次和项目一致，仅来源页面不同"
}
```

`confidence` 仅允许 `high`、`medium`、`low`。定时任务自动处理 high/medium；low 必须进入待确认摘要。
