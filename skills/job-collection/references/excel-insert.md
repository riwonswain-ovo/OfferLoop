# 飞书多维表格写入规范

> 每次来源同步后自动将结果保存到飞书多维表格，写入前必须读取本文件。

## 表格列定义（22 列）

| # | 列名 | 说明 |
|---|---|---|
| 1 | 编号 | 序号，追加时接续已有最大编号 |
| 2 | 信息更新时间 | DateTime 字段，记录该条招聘信息在来源表里的更新时间。没有可靠日期时留空；主表 + 对应子表必须写同一值 |
| 3 | 投递进度 | 单选，三态：`待确认` / `已投递` / `已拒绝`。详见下方「投递进度字段说明」 |
| 4 | 公司 | 公司名称 |
| 5 | 届次 | MultiSelect，2026届 / 2027届 / 2028届（与其他届次取值动态追加） |
| 6 | 招聘批次 | SingleSelect，14 取值见下方「招聘批次取值」。**必须单值**——多批次必须拆多行，详见下方「字段标注规则」 |
| 7 | 招聘项目 | 招聘项目名称（有就填，没有留空）。互联网公司常有细分项目，如京东有 JDS、JDY、TET；阿里有星图、青藤、阿里星；腾讯有技术大咖、产品培训生。需在企业官网或招聘页确认后填写，不要凭印象编造 |
| 8 | 招聘岗位 | 该企业本批次招的主要岗位类别，多个用 `、` 分隔（如「Java 开发、产品经理、管培生」）。公告里没明确岗位类别的留空 |
| 9 | 城市 | 工作地点，多个用 `、` 分隔 |
| 10 | 投递截止时间 | **Text 字段**——可写 ISO 日期 `2026-08-31`，也可写 `尽快投递` / `招满即止` / `未公布`（用 DateTime 字段无法承载灵活文案，故改 Text） |
| 11 | 学历要求 | 学历门槛，如 `本科` / `硕士` / `博士` / `本科及以上` / `硕士及以上`。公告里没写明留空 |
| 12 | 专业要求 | JD 里的专业对口要求，多个用 `、` 分隔（如「经济学、金融学、数学」）。公告里没写明留空 |
| 13 | 行业标签 | 多选，取值见下方「行业标签取值」——从 `field-contract.md` 1.3 节的 13 个 `industry_module` ID 映射为中文名 |
| 14 | 是否笔试 | MultiSelect，选项 `要笔试` / `不要笔试` / `未知`。公告明写需笔试 → `要笔试`；明写免笔试 → `不要笔试`；没写 → `未知` |
| 15 | 来源平台 | 用户登记的信息源名称 |
| 16 | 公告链接 | **Url 字段**——来源表中记录的公告页面 URL，用于回溯这条信息是从哪个页面看到的 |
| 17 | 投递链接 | **Text 字段**——网申入口 URL 或投递方式（如「邮箱投递：xxx@xx.com」）。注意是 Text 不是 Url，因为部分投递入口是非 URL 形式（邮箱 / 二维码说明） |
| 18 | 内推链接 | 走员工推荐通道的投递入口 URL，可空（仅内推表/推文里带就填，公开官方渠道搜来的常为空） |
| 19 | 内推码 | 内推码字符串，可空（同上，跟内推链接配对出现） |
| 20 | 备注 | Text 字段，抽取时拿不准的额外信息（如「需现场投递」「只招本地」「需 onsite 笔试」等），可空 |
| 21 | 企业性质 | SingleSelect，5 取值：互联网 / 金融银行 / 外企 / 央国企 / 其他私企。按 field-contract.md 1.6 节优先级判定（互联网 > 金融银行 > 外企 > 央国企 > 其他私企） |
| 22 | 子表 record_id | Text 字段，存该条线索在对应子表里的飞书 record_id。用户可隐藏此列，投递进度同步时 AI 用它定位子表行 |

> **字段名优先原则**：除上方表格列定义外，所有写入、更新、回填、去重、视图配置都按字段名执行，不按列号执行。
>
> **「公告链接」与「投递链接」是两套语义**：「公告链接」是来源表记录的公告页面 URL，「投递链接」是真正投递入口。两者可同时存在，也可只存在一个。
>
> **「内推链接」与「公告链接」/「投递链接」又不同**：内推链接是走员工推荐通道的入口，跟公开网申入口平行存在，不要混到一起。
>
> **去重判定只用「投递链接」**——「公告链接」是回溯用、「内推链接」是回填用，都不触发新增或去重判断。详见 `references/dedup_judge.md`。

### 投递进度字段说明

`投递进度` 列只有三种取值，承载本 skill 的整个跟进闭环：

| 投递进度 | 含义 | 何时填 |
|---|---|---|
| `待确认` | 默认状态。来源同步写入后立即标这个 | 任何新条目写入主表时，「投递进度」字段直接填 `待确认` |
| `已投递` | 用户已对该企业发起投递 | 用户在对话里说"投了 XX""已经投了"等，AI 把对应行的状态改为 `已投递` |
| `已拒绝` | 用户明确不投该企业 | 用户在对话里明确说"不要 XX""拒绝了 XX""XX 不投"等，AI 改投递进度为 `已拒绝` |

### 推送与状态联动规则（双向同步：主表 ↔ 子表）

**这是「投递进度」字段存在的核心理由——它驱动次日推送逻辑**：

1. **当天来源同步完成双写主表 + 子表后**：所有新条目在主表 + 子表的投递进度都默认 `待确认`。
2. **第二天（及以后每次用户重新触发 skill）**：先跑「入口 B 同步检测」（用户可能手动改过投递进度），再扫描主表，把所有 `投递进度 = 待确认` 的企业整理成"待确认清单"再次推送给用户，提醒用户逐条决定要不要投。
3. **每次推送完待确认清单后**：**主动问用户**「以上 N 家里，有哪些是你明确不投的？告诉我我才好把它们标成『已拒绝』，之后就不会再重复提醒了。」
4. 用户回复 → AI 按「入口 A 流程」改投递进度（主表 + 子表双向同步）：用户点名的企业投递进度改为 `已拒绝`，用户说"投了"的改为 `已投递`，其余继续保留 `待确认` 等下一轮再问。
5. `已投递` 和 `已拒绝` 的条目**不再出现在后续推送里**，但仍保留在主表 + 子表中以便回溯。

#### 投递进度变更的两种入口

- **入口 A：用户在 0.5 步推送里点某条改投递进度** → AI 调 API 改主表 + 子表
- **入口 B：用户在飞书界面（主表或子表）手动改投递进度** → 飞书自动化 / webhook / event listener 实时或准实时同步，skill 启动时再做兜底一致性校验

#### 入口 A 流程（AI 主动改，0.5 步推送后用户回复时触发）

**前置：批次消歧（必须先做，再走下方 1-4 步）**

用户在 0.5 步推送后说"投了 X" / "不要 X" 时，X 在主表里可能有多行（同公司多批次）。**必须先消歧，再改投递进度**：

1. 按公司名 X 在主表里捞所有相关行（含所有批次）
2. 只有 1 行 → 直接进下方 1-4 步改投递进度
3. 有 ≥2 行 → **必须列出所有行让用户点哪几行**，模板见 `SKILL.md` 第零点五步第 3 步。用户回编号（如「1, 3」）或「全部」后才进下方 1-4 步逐行改投递进度
4. **禁止擅自假设**——不要默认"投了字节跳动"就是全部批次都投了，必须用户明确点选后才改

**改投递进度操作（按消歧结果，对每条要改的行逐条执行）**

1. 按主表 record_id 调 `update_record` 改主表「投递进度」列
2. 从主表该行读「子表 record_id」列 → 拿到对应子表的 record_id
3. 按子表 record_id 调 `update_record` 改子表「投递进度」列
4. 子表定位哪个 sheet：先从主表该行读「企业性质」列知道是哪类子表，再按子表 table_id 映射定位

#### 入口 B 流程（用户手动改、实时/准实时同步 + 启动兜底）

目标：人类只要在任一位置（主表、主表分类视图、子表、子表分类视图）手动修改 `投递进度`，对应记录都要跟着同步。主表分类视图本质是主表记录视图，会自然同步；真正需要 API 同步的是主表记录与物理子表记录。

1. **实时/准实时层**：优先用飞书自动化 / webhook / event listener 监听主表和 5 个子表的 `投递进度` 变更。
2. 根据 record_id 映射定位对应记录：
   - 主表「子表 record_id」存对应子表 record_id。
   - 子表「子表 record_id」存对应主表 record_id。
3. 幂等更新：
   - 目标记录 `投递进度` 已相同 → 不写。
   - 目标记录不同 → 只更新 `投递进度` 字段，不全行重写。
4. 循环保护：
   - webhook/自动化被同步写入再次触发时，因目标值已相同会直接跳过。
   - 如平台支持事件来源标识，标记 AI 同步写入并在下一次事件中忽略同一 pair。
5. **启动兜底层**：每次 skill 启动时读取主表 + 5 个子表的必要字段（record_id、子表 record_id、投递进度、modified_at），做全量或最近变更一致性校验，修复漏掉的事件。不能只检测主表，因为用户可能只在子表里改过投递进度。
6. 检测差异：
   - 主表与子表投递进度不一致 → 以最近 `modified_at` 为准，把旧的那个改成新的。
   - 两边修改时间完全相同 → 默认以主表为准；如果上下文显示存在真实冲突，在对话里让用户确认。
   - 只有一边存在 record_id → 进入「双写一致性校验」的孤儿记录处理，不要随便删。

#### 飞书 Base workflow 落地要求（必须创建，不只写规则）

首次建表或修复已有表时，必须给 Base 创建并启用 10 条投递进度同步 workflow：

- `企业清单 → 互联网 / 金融银行 / 外企 / 央国企 / 其他私企`：5 条。
- `互联网 / 金融银行 / 外企 / 央国企 / 其他私企 → 企业清单`：5 条。

每条 workflow 的核心结构：

1. `SetRecordTrigger` 监听源表字段 `投递进度`。
2. `SetRecordAction` 更新目标表字段 `投递进度`。
3. 目标记录定位方式固定为：目标表 `子表 record_id == 触发记录 recordId`。
4. 写入值使用触发记录的 `投递进度` 字段 ref。注意每张表的 `投递进度` field_id 不同，创建 workflow 前必须分别 `field-list` 读取真实 field_id，不能复用主表 field_id。
5. workflow 创建后默认 disabled，必须显式 `+workflow-enable`。创建完后 `+workflow-list` 验证 10 条 workflow 都是 `enabled`。
6. `trigger_control_list` 必须排除自动化产生的更新事件（当前飞书配置使用 `pasteUpdate`、`syncUpdate`），避免 A → B 后又由 B → A 无限触发；`max_set_record_num` 固定为 1。
7. 启用后做一次烟测：任选一条已配对记录，子表改 `投递进度` → 等主表同步；再从主表改回原值 → 等子表同步；最后跑全表一致性审计确认 `mismatch_count = 0`。
8. workflow 只更新底层配对记录，不向任何视图“新增一行”。主表和子表状态一致后，飞书会按过滤条件自动让记录进入或退出各自的 `已投递` 视图。

已验证的 workflow 形态示例（伪 JSON，字段 ID 必须按真实表替换）：

```json
{
  "title": "投递进度同步：互联网 → 主表",
  "steps": [
    {
      "id": "tr_internet_status",
      "type": "SetRecordTrigger",
      "next": "set_main",
      "data": {
        "table_name": "互联网",
        "field_watch_info": [{"field_name": "投递进度"}],
        "trigger_control_list": ["pasteUpdate", "syncUpdate"],
        "condition_list": null
      }
    },
    {
      "id": "set_main",
      "type": "SetRecordAction",
      "next": null,
      "data": {
        "table_name": "企业清单",
        "max_set_record_num": 1,
        "filter_info": {
          "conjunction": "and",
          "conditions": [
            {
              "field_name": "子表 record_id",
              "operator": "is",
              "value": [{"value_type": "ref", "value": "$.tr_internet_status.recordId"}]
            }
          ]
        },
        "field_values": [
          {
            "field_name": "投递进度",
            "value": [{"value_type": "ref", "value": "$.tr_internet_status.<该子表投递进度field_id>"}]
          }
        ]
      }
    }
  ]
}
```

视图里的记录不需要单独同步：主表分类视图是主表记录的不同展示，子表分类视图是子表记录的不同展示。workflow 覆盖主表实体记录和物理子表实体记录后，所有视图会自然跟随。

#### 单边写入失败的修复

若历史某次写入时主表成功但子表失败（或反过来），主表「子表 record_id」列会为空或指向不存在的子表行。下次触发 skill 检测到这类残缺记录时：

- 主表有 record_id 但子表没对应行 → 按主表数据补写一条到对应子表，回填主表「子表 record_id」
- 子表有行但主表「子表 record_id」列为空 → 按子表 record_id 反查定位子表行，回填主表「子表 record_id」列
- 子表 record_id 指向的子表行不存在（被用户误删）→ 按主表数据重新写一条到子表，更新主表「子表 record_id」

### 最小可执行示例：读表 → 筛待确认 → 提示用户 → 改投递进度（含双向同步）

伪代码（实际调飞书 API 时按当前文档为准）：

```python
# 1. 读主表全部记录
main_records = feishu_bitable_list_records(app_token, main_table_id)

# 2. 入口 B 同步检测：list 5 个子表，按 record_id 配对，按 modified_at 同步投递进度差异
sub_tables = {"互联网": table_id_1, "金融银行": table_id_2, ...}  # 5 子表映射
for sub_name, sub_tid in sub_tables.items():
    sub_records = feishu_bitable_list_records(app_token, sub_tid)
    # 按 record_id 配对，比对投递进度差异，以 modified_at 为准双向同步
    sync_status_if_diff(main_records, sub_records, sub_tid)

# 3. 筛出主表 投递进度=待确认 的条目（只取主表那份，避免双写导致清单重复）
pending = [r for r in main_records if r.get("投递进度") == "待确认"]

# 4. 整理成清单推给用户（按 SKILL.md 第零点五步格式输出）
print(f"📋 待确认清单（共 {len(pending)} 家）")
for i, r in enumerate(pending, 1):
    print(f"{i}. {r['公司']} - {r['招聘批次']}（{r['城市']}, 截止 {r['投递截止时间']}）")

# 5. 等用户回复后改投递进度（用户说"1, 5 不投" → 入口 A 流程：主表 + 子表双向同步）
def update_status双向(main_record_id, sub_record_id, sub_table_id, new_status):
    # 先改主表
    feishu_bitable_update_record(app_token, main_table_id, main_record_id, {
        "投递进度": new_status   # SingleSelect 字段直接传字符串
    })
    # 再改子表
    feishu_bitable_update_record(app_token, sub_table_id, sub_record_id, {
        "投递进度": new_status
    })

# 6. 投递进度更新只改「投递进度」列，不要全行重写
```

注意：
- 读表时按 `投递进度` 字段值 == 字符串 `"待确认"` 过滤即可，飞书 SingleSelect 字段返回的就是字符串
- update 只改「投递进度」字段，按 `record_id` 定位行；不要全行重写以免擦掉其他字段
- 不读取 `image` / 文件类字段，主表里没有这类字段

## 字段标注规则

### 投递截止时间

- 页面明确写「2026-08-31」→ 填 `2026-08-31`
- 页面写「招满即止」「常年招聘」→ 填 `招满即止`
- 页面写「尽快投递」「先到先得」→ 填 `尽快投递`
- 查不到截止时间 → 填 `未公布`

> 该字段是 Text 字段，可承载灵活文案，不再是 DateTime。

### 招聘批次

- **字段类型 SingleSelect，必须单值**——一行只承载一个(公司, 批次)组合，多批次必须拆多行（其他字段复制不变、批次填单值）
- **预置 14 取值**（建表时配进 `property.options`，按以下顺序）：
  - `暑期实习` / `暑期实习生` / `暑期`
  - `秋招提前批` / `提前批`
  - `秋招` / `秋季校园招聘` / `秋招正式批` / `秋招专场`
  - `春招` / `春季校园招聘`
  - `补录` / `春招补录` / `秋招补录` / `校招补录`
- 允许动态追加：抽取时遇到上述 14 个之外的批次措辞（如「冬季招聘」「专项招聘」），先调 `+field-update` 把新值加进 `property.options`，再写入该值——不要强行归到 14 个里
- 多批次写成一行（如"暑期实习、秋招提前批"）是**禁止的写法**——会让后续投递进度变更无法消歧（用户说"投了字节跳动"时 AI 不知道投的是哪个批次），写入前必须按 `references/personal-excel-source.md` 第 2.1 节拆开
- 该约束是 `dedup_judge.md` 环节 B dedup 规则"按 `公司 + 招聘批次` 判重"的隐含前提——若一行多批次，dedup 键就退化成只看公司名，会把不同批次当成重复错杀
- 源数据是一行多批次（用户上传的飞书表常见此情况）→ 按 `references/personal-excel-source.md` 第二节的拆行规则处理后再写入
- **视图过滤依赖此字段为 SingleSelect**——下方"视图建视图"节的 6 个批次视图用 `intersects` 多值精确匹配，要求字段类型是 SingleSelect 才能传数组值列表

### 届次

- 抽取时根据公告里的「2027届」「2026届校招」等措辞判定，命中哪个就标哪个
- 公告没明确写届次但写了批次 → 推算：搜暑期实习/提前批/秋招 → 下一届；搜春招/补录 → 当前届（推算规则见 `SKILL.md` 第三步「届别规则」）
- 推算也拿不准 → 留空，不要瞎填

### 招聘岗位

- 公告里明确写了岗位类别（如「Java 开发 / 产品经理 / 管培生」）→ 全部填入，多个用 `、` 分隔
- 公告只写「详见岗位列表」「点击查看全部岗位」没列具体岗位 → 留空
- 不要凭公司主营业务猜岗位（互联网公司不等于只招研发，央国企也招管培生）

### 学历要求

- 公告明写「本科及以上」「硕士及以上」「博士」→ 直接填
- 公告写「全日制本科」「研究生」→ 直接填公告原文措辞
- 公告没写学历要求 → 留空

### 专业要求

- 公告 JD 里的专业对口要求（如「经济学、金融学、数学、统计学」）→ 多个用 `、` 分隔
- 公告写「专业不限」→ 填 `专业不限`
- 公告没提专业要求 → 留空

### 是否笔试

- 公告明写「需笔试」「笔试环节」→ `要笔试`
- 公告明写「免笔试」「无笔试」→ `不要笔试`
- 公告没提 → `未知`

### 备注

- 抽取时拿不准的额外信息（如「需现场投递」「只招本地户口」「需 onsite 笔试」「接受往届生」等）填这里
- 公告里没有额外约束 → 留空
- 不要把主字段已经记录的信息重复填到备注里

### 行业标签

- 每条企业线索**必须至少打一个标签**，不要留空
- 优先用 13 个行业 ID 之一；检索结果企业明显不属于这 13 类（如律所、咨询机构但不在 `marketing_consulting` 范畴、政府机关等）→ 打 `其他`
- 同一企业跨多个行业（如比亚迪既造车又做电池）→ 可多选
- 标签优先读取来源记录已提供的行业字段；来源未提供或无法映射时 → AI 根据公司主营业务判一个

## 追加模式（双写主表 + 子表，每次同步后自动执行）

每次 skill 跑完来源读取、标准化、去重、硬筛和质检后，**自动按本节流程把保留的企业线索双写进飞书主表 + 对应子表**。

「追加」= 只把表里还没有的新条目加进去，不是整表重写。「双写」= 同一条线索按企业性质判定后，写主表 + 写对应子表（两份数据）。

具体步骤：

1. **读取主表 + 5 个子表全部行**，主要拿四个字段：`编号`、`公司`、`招聘批次`、`投递链接`。子表按企业性质分类，读 5 个子表是为了跨子表检测同企业同批次是否已存在
2. **算下一个起始编号**：当前主表里最大的编号 +1。例如表里已有 1-95 条，新行从 96 开始编
3. **对每条新条目逐一判断要不要加进去**（去重，避免重复加）：
   - 新条目的 `投递链接` 跟主表某行 `投递链接` 一模一样 → **这条不加**，跳过
   - 新条目的 `公司 + 招聘批次` 跟主表某行一模一样，但投递链接不同 → 走 `references/dedup_judge.md` 的 LLM 复判流程，复判结论是"重复"→ 不加（旧条保留），结论是"不重复"→ 加
   - 主表里完全没有这家公司的这条批次 → **直接加**
   - **跨表去重规则**：主表与子表之间不做去重（同条线索双写是设计意图，不是重复）。只在主表内部 + 子表内部各自判定，详见 `references/dedup_judge.md` 环节 B
4. **判定 enterprise_type**：按 `field-contract.md` 1.6 节优先级规则判定该条线索的企业性质（互联网 > 金融银行 > 外企 > 央国企 > 其他私企，命中即归），拿到 5 取值之一
5. **先写主表**：调 `batch_create` 或 `insert_records` 写入主表 22 列（含「企业性质」列填判定结果、「子表 record_id」列暂空；若来源有更新时间则填「信息更新时间」），拿到主表 record_id
6. **再写子表**：按判定的 enterprise_type 找到对应子表 table_id，调 `batch_create` 写入该子表 22 列（「子表 record_id」列填主表 record_id；「信息更新时间」与主表一致），拿到子表 record_id
7. **回填主表**：调 `update_record` 把主表该行的「子表 record_id」列填上子表 record_id。这样主表 ↔ 子表的 record_id 双向映射就建好了，投递进度同步时双向都能直接定位
8. **新条目投递进度默认 `待确认`**——主表 + 子表的「投递进度」字段都直接填 `待确认`，这一步和上方"投递进度字段说明"对齐
9. **回填场景：内推字段 update**——若 `dedup_judge.md` 环节 B 判定"新条目跟旧条目是同一条，新条目只多带了 `referral_url` / `referral_code`"，不新加行，调飞书 API 旧行的 update（按 `lead_id` 定位行）把内推字段补进去：
   - 仅当旧行的 `内推链接` / `内推码` 列为空时补；旧行已有值 → 不覆盖（保留旧值，避免冲突）
   - update 用字段级更新（只改这两列），不要全行重写
   - 同样适用于「公告链接」回填——新条目带的 `公告链接` 是旧条目缺的来源 URL，可按同样规则回填到「公告链接」字段（仅当旧行「公告链接」为空时补）
   - 内推回填**只在主表做**——子表是分类视图，回填到主表即可；下次触发 skill 时若主表有内推字段而子表没有，按双向同步规则同步子表
10. **追加完告诉用户**：本次新增了几条（主表+子表各几条）、回填了几条内推信息、跳过了几条重复、最终主表里现在共多少条
11. **写入后 sanity check**——主表 + 子表的 batch_create 返回成功后立刻调 list API 读回本批新写入记录（数据量很大时至少抽样每个目标子表 1 条 + 主表最新 3 条），验证：
   - 单选下拉「投递进度」「企业性质」字段值是否真的是 `待确认` / 对应企业性质名（之前出现过写整数导致字段值变成空数组）
   - 多选「行业标签」「届次」「是否笔试」字段值数组是否含正确的中文名（而不是 option_id 之类）
   - 「编号」字段是否被存成字符串而非整数
   - 「公告链接」Url 字段是否真的存成可点击 URL（而不是被截成纯文本）
   - 主表「子表 record_id」列是否填上了子表 record_id
   - 若新条目来自用户上传飞书源表，主表 + 子表「信息更新时间」是否都写入源表「更新时间」对应日期
   - 任意一项不对 → 调 update API 修正该字段后再告用户成功；不要写入成功就当通过
12. **视图排序验收（最新置顶，强制）**——写表后立即对主表 + 5 子表的所有 grid 视图读回 sort；不是 `信息更新时间 desc, 编号 desc` 就补 `+view-set-sort` 并再次读回确认。飞书记录物理顺序永远是追加到底部，用户看到“最新在最上方”只能靠视图排序，不能跳过。
13. **写表后一致性校验（强制）**——sanity check 通过后立即跑「子表归类、子表重复、主子表数量与映射」检查，任一不通过主动修复，详见本文件末尾「写表后一致性校验」节。
14. **验收失败禁止报成功**——日期缺失、视图排序未生效、主表孤儿行、子表孤儿行、同编号同公司同批次重复行任一存在时，必须先修复或明确告诉用户未完成。

**写入顺序很关键**：必须先主表后子表，因为子表写入时要把主表 record_id 填进子表的「子表 record_id」列；然后再回填主表的「子表 record_id」列。

举例：
> 用户昨天主表里有 80 条。今天两个已登记来源产生 20 条候选，其中 5 条投递链接已存在、3 条公司+批次复判为重复。最终主表追加 12 条，对应子表也各追加，编号 81-92 是今天新增的，状态都是 `待确认`。

## 5 个子表建表（主表建完后立即建）

主表建完后，在同一 base 下再建 5 个 sheet，每个子表字段结构与主表完全一致（22 列，含「企业性质」「子表 record_id」「信息更新时间」——子表里这些列可保留作字段对齐，写入时「企业性质」填对应值、「子表 record_id」填主表 record_id，「信息更新时间」与主表一致）。

**子表命名**：`互联网` / `金融银行` / `外企` / `央国企` / `其他私企`

**建表 API**：`POST /open-apis/bitable/v1/apps/{app_token}/tables`（同一 base 下创建新 sheet）

**请求体**（每个子表调一次，共 5 次）：

```json
{
  "table": {
    "name": "互联网",
    "default_view_name": "默认视图",
    "fields": [
      {"field_name": "编号", "type": 1},
      {"field_name": "信息更新时间", "type": 5, "ui_type": "DateTime", "property": {"date_formatter": "yyyy/MM/dd"}},
      {"field_name": "投递进度", "type": 3, "ui_type": "SingleSelect", "property": {"options": [
        {"name": "待确认"}, {"name": "已投递"}, {"name": "已拒绝"}
      ]}},
      {"field_name": "公司", "type": 1},
      "... 其余字段复用主表 22 字段 payload，完整粘贴主表建表 payload 的 fields 数组"
    ]
  }
}
```

> 子表字段结构与主表完全一致，包括「企业性质」(SingleSelect 同 5 选项) 和「子表 record_id」(Text) 都建上——字段对齐方便后续投递进度同步按字段名直接读写。

**返回值**：每个子表创建后会返回 `table_id`，记录 5 个 `table_id` + 子表名映射，存进对话上下文（不持久化到 .env——下次触发时按 base app_token 重新 list tables 拿回）。

**建表后清理占位行**：跟主表一样，每个子表建完后立刻清理空行（见下方"建表后清理占位行"节同款流程）。

## 视图建视图（默认视图 + 6 个批次视图 + 已投递，主表 + 5 子表共 48 个 grid 视图）

主表 + 5 子表建完、占位行清理完后，**保留默认 grid 视图，并在每个 sheet 里建 6 个批次 grid 视图和 1 个 `已投递` grid 视图**。批次视图按「招聘批次」字段过滤；`已投递` 视图按「投递进度」过滤；全部视图都必须设置排序，让用户打开表时最新信息在最上方。

**时机**：建完 6 sheet → 清理占位行 → **建 36 个批次视图 + 6 个 `已投递` 视图 + 给全部 48 个视图设置排序** → 创建并启用 10 条投递进度双向 workflow → 加协作者。视图过滤是动态的，新写入或状态变化的记录无需手动归视图。

### 6 个批次视图定义

每个 sheet 建以下 6 个批次视图，过滤规则用 `intersects` 多值精确匹配（值列表互斥，一条记录只进一个视图）：

| 视图名 | 过滤方式 | 值列表 |
|---|---|---|
| `暑期实习` | `intersects` | `["暑期实习", "暑期实习生", "暑期"]` |
| `秋招提前批` | `intersects` | `["秋招提前批", "提前批"]` |
| `秋招` | `intersects` | `["秋招", "秋季校园招聘", "秋招正式批", "秋招专场"]` |
| `春招` | `intersects` | `["春招", "春季校园招聘"]` |
| `补录` | `intersects` | `["补录", "春招补录", "秋招补录", "校招补录"]` |
| `其他批次` | `disjoint` | 上述 5 类所有值的并集（即全部 14 个预置值，取反） |

> **值列表互斥 → 无需优先级规则**：14 个预置值按批次类型分到 5 个 `intersects` 视图里，每个值只属于一个视图的值列表。一条记录的「招聘批次」是单值，命中哪个视图的值列表就进哪个视图，不会同时进多个。`其他批次` 用 `disjoint` 取反兜底——「招聘批次」值不在 14 个预置里时（动态追加的新批次）进这个视图。

### 建视图命令（每个 sheet 跑 6 次批次视图 + 1 次已投递视图）

对主表 + 5 子表共 6 个 sheet，每个 sheet 跑 7 次「建视图 + 设过滤」组合（共 42 次 view-create + 42 次 filter 配置）。建完后，对每个 sheet 的默认视图、6 个批次视图和 `已投递` 视图都设置同一套 sort：`信息更新时间` 降序，其次 `编号` 降序。

### `已投递` 视图定义与列顺序

- 视图名固定为 `已投递`，过滤条件固定为 `投递进度 = 已投递`。
- 创建后先用 `+view-get-visible-fields` 读取该表默认视图的可见字段，再把返回列表原样传给 `+view-set-visible-fields`；不能跨表复用 field_id，也不能硬编码主表字段数。主表通常比子表多 `父记录` 字段。
- 读回 `已投递` 视图确认字段集合和顺序均与默认视图一致。若接口因集合相同而返回 `no operation produced`，但读回顺序仍错误，先临时设置为前 1 到 2 个字段，再恢复完整字段列表并再次读回。
- `已投递` 是过滤视图，不是物理子表。禁止向该视图单独复制或创建记录。

推荐优先使用当前 `lark-cli` 的 `+view-set-filter`、`+view-set-sort` 和 `+view-set-visible-fields`，每次写后都读回验证。只有 CLI 不支持当前服务端参数时，才退回原生 PATCH。

> **K7 卡点（重要）**：旧版文档中的 `{"logic":"and","conditions":[[字段名,operator,value]]}` 三元数组 payload 不可用。当前优先调用 `lark-cli base +view-set-filter`，conditions 必须使用对象结构，并在调用后读回单个视图确认过滤已落地。若当前 CLI 与服务端版本不兼容，再使用 `PATCH /bitable/v1/apps/{app_token}/tables/{table_id}/views/{view_id}`；raw API 的过滤配置位于 `property.filter_info`，不是 `filter`。

```bash
# 1. 建视图（grid 类型）
curl -s -X POST \
  "https://open.feishu.cn/open-apis/bitable/v1/apps/${APP_TOKEN}/tables/${TABLE_ID}/views" \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"view_name":"暑期实习","view_type":"grid"}'
# 返回 data.view.view_id 记下来，下一步要用

# 2. PATCH 设 filter_info（用上一步返回的 view_id）
curl -s -X PATCH \
  "https://open.feishu.cn/open-apis/bitable/v1/apps/${APP_TOKEN}/tables/${TABLE_ID}/views/${VIEW_ID}" \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "filter_info": {
      "conjunction": "and",
      "conditions": [
        {"field_name":"招聘批次","operator":"intersects","value":["暑期实习","暑期实习生","暑期"]}
      ]
    }
  }'
```

`秋招` 视图示例（只列 PATCH body，建视图命令同上换 view_name）：
```json
{
  "filter_info": {
    "conjunction": "and",
    "conditions": [
      {"field_name":"招聘批次","operator":"intersects","value":["秋招","秋季校园招聘","秋招正式批","秋招专场"]}

  }
}
```

`其他批次` 视图用 `disjoint` 取反：
```json
{
  "filter_info": {
    "conjunction": "and",
    "conditions": [
      {"field_name":"招聘批次","operator":"disjoint","value":["暑期实习","暑期实习生","暑期","秋招提前批","提前批","秋招","秋季校园招聘","秋招正式批","秋招专场","春招","春季校园招聘","补录","春招补录","秋招补录","校招补录"]}
    ]
  }
}
```

### 视图排序规则（最新信息置顶）

所有 48 个 grid 视图（每张表的默认视图 + 6 个批次视图 + `已投递`）都必须设置排序：

```json
{
  "sort_config": [
    {"field": "信息更新时间", "desc": true},
    {"field": "编号", "desc": true}
  ]
}
```

推荐使用 `lark-cli base +view-set-sort`：

```bash
lark-cli base +view-set-sort \
  --base-token "${APP_TOKEN}" \
  --table-id "${TABLE_ID}" \
  --view-id "${VIEW_ID}" \
  --json '{"sort_config":[{"field":"信息更新时间","desc":true},{"field":"编号","desc":true}]}' \
  --as user
```

这只是视图排序，不改变物理写入顺序，也不影响主表 ↔ 子表 record_id 映射。不要为了“最新在上”去重排记录或重写编号；新记录仍按最大编号 +1 追加，展示层靠视图排序置顶。

> **字段名 vs 字段 ID**：conditions 里的 `field_name` 优先传字段名（中文即可），飞书 API 会按字段名解析。字段名含特殊字符或跨表重名时才改用 `field_id`（key 名也跟着改成 `field_id`）。
>
> **curl 超时**：建表 / 建视图偶发慢请求，curl 默认 30s 不够会「服务端已建成功但 curl 返回空」，重试就会建重。统一加 `--max-time 90`，curl 返回空时**先调 `GET /tables/{table_id}/views` 查现有视图列表**确认是否已建成功，再决定是否重试。

### 前置依赖：批次值必须归一到预置列表

视图过滤生效的前提是「招聘批次」字段值**精确等于**预置 14 个值之一。写入前必须按上方"字段标注规则 → 招聘批次"节归一化：

- 抽取 / 源表映射时遇到 `暑期实习生招聘` / `暑期实习招聘` → 归到 `暑期实习`
- 遇到 `秋招正式批` / `秋招专场` → 保留原措辞（已在预置列表里）
- 遇到 14 个之外的全新批次措辞（如 `冬季招聘` `专项招聘`）→ 先调 `+field-update` 把新值加进 SingleSelect 的 `property.options`，再写入该值；同时把新值追加到 `其他批次` 视图的 `disjoint` 值列表里（否则新值会同时进 `其他批次` 和某个 `intersects` 视图，过滤失效）

### 建视图失败的兜底

- `POST /views` 失败（飞书限流 / 权限不足）→ 重试 1 次，仍失败跳过该视图继续建下一个，最后告知用户「X 个视图建失败，可手动在飞书界面按以上过滤条件新建」
- `PATCH /views/{view_id}` 返回 404 或 `code=9499`（path 或 field 名错）→ 检查 url 是不是 `/bitable/v1/apps/{app_token}/tables/{table_id}/views/{view_id}`、body 顶层 key 是不是 `filter_info`（不是 `filter`）、conditions 是不是对象数组（不是三元数组）。这三处任一写错都报这类错
- `PATCH` 返回 operator 不支持（`1254101` 之类）→ 检查「招聘批次」字段是否真的是 SingleSelect（建表时若误建成 Text 会报 operator 不支持 `intersects`/`disjoint`），修字段类型后重试
- curl 返回空字符串 → 服务端可能已建成功但 curl 超时。**先调 `GET /tables/{table_id}/views` 查视图列表**确认是否已存在，已存在就复用 view_id 进下一步 PATCH，不要重试 POST 避免建重

### 后续触发：幂等修复，不建重复视图

视图首次建表时创建，后续触发时必须检查并幂等修复：
- 先按视图名查找；视图已存在 → 复用 view_id，不重建
- `已投递` 视图缺失 → 创建并补过滤、排序、列顺序；同名存在多个 → 停止自动删除并在验收摘要报告重复，避免误删用户视图
- 任一 grid 视图缺少标准排序 → 补 `信息更新时间 desc, 编号 desc`
- `已投递` 过滤或列顺序漂移 → 修复为标准配置并读回
- 用户在飞书界面手动改了视图过滤 → 不主动覆盖，除非用户明确说"把视图过滤改回标准配置"
- 新追加的批次值落到 `其他批次` 视图里 → 用户看到后若想单独建视图，按上方"前置依赖"节加新值 + 新视图

## 飞书多维表格操作要求

- 字段顺序按上方表头定义，不要随意调换
- 「公告链接」列优先用 **Url 字段类型**（`type=11 + ui_type="Url"`）；若当前 Base 配置写入 Url 失败，按故障恢复章节降级为 Text
- 「投递链接」列用 **Text 字段类型**——`type=1`。保留 Text 是因为部分投递入口是非 URL 形式（「邮箱投递：xxx@xx.com」「见原文二维码」等），用 Url 字段会拒非 URL 内容
- 「内推链接」列用 Text 字段类型（`type=1`）——内推链接也常出现非标准 URL 形式
- 「投递截止时间」列用 **Text 字段类型**（`type=1`）——不再是 DateTime，要承载 `尽快投递` / `招满即止` / `未公布` 等灵活文案
- 「招聘批次」列用 **SingleSelect 字段类型**（`type=3 + ui_type="SingleSelect"`），预置 14 个选项（取值见上方"字段标注规则 → 招聘批次"节）。SingleSelect 是为了支持下方"视图建视图"节用 `intersects` 多值精确匹配做批次过滤——Text 字段的 `intersects` 只接受单个字符串，无法实现"值列表任一命中"
- 「届次」列用 **MultiSelect 字段类型**（`type=4 + ui_type="MultiSelect"`），预先把 `2026届` / `2027届` / `2028届` 配进 `property.options`，飞书 MultiSelect 支持后续动态追加选项
- 「行业标签」列用 **MultiSelect** 字段类型——`type=4 + ui_type="MultiSelect"`，**不要用 `type=7` Checkbox**（会报 `1254085 CheckboxFieldPropertyError`）。预先把 14 个取值配进 `property.options` 数组，每个 option 形如 `{"name": "互联网与科技"}`
- 「是否笔试」列用 **MultiSelect 字段类型**（`type=4 + ui_type="MultiSelect"`），预置 `要笔试` / `不要笔试` / `未知` 三个选项
- 「投递进度」列用 **SingleSelect 字段类型**（`type=3`），预先把 `待确认` / `已投递` / `已拒绝` 三个选项配进 `property.options`，默认值设为 `待确认`
- 「编号」列用 Text 字段类型（`type=1`）——**必须传字符串 `"1"` 不能传整数 `1`**，整数会报 `1254060 TextFieldConvFail`
- 「内推码」列用文本字段类型（`type=1`）
- 「招聘岗位」「学历要求」「专业要求」「备注」列均用 Text 字段类型（`type=1`）
- 不要把表头涂色或加 Excel 样式——飞书多维表格不是 Excel，不使用单元格底色 / 字体颜色 / 列宽数组

## 加协作者流程（建表后立即执行，不是写完表后）

机器人用 `tenant_access_token` 调 `POST /bitable/v1/apps` 建出来的 base，**owner 默认是机器人 App 自己**，用户本人对这张表没有任何默认权限——必须建表后立即加用户为协作者才能打开。

### 时机

**7 个 sheet 建完后立即执行**（`init-workflow.md` 4.1 节第 5 步）。**不是写完表后**——拖到写完才加的话，用户在搜索期间想看表都看不到。

### API

```
POST /open-apis/drive/v1/permissions/{app_token}/members?need_notification=true&type=bitable
Authorization: Bearer {tenant_access_token}
Content-Type: application/json

{
  "member_type": "email" | "openid",
  "member_id": "{用户在 2.6 节给的标识（手机号需先转 openid）}",
  "perm": "full_access"
}
```

`perm` 取值：`view` / `comment` / `edit` / `full_access`。本 skill 用 `full_access`——用户可编辑 + 可转交 + 可删，跟表 owner 等价。

> **K8 卡点（重要）**：`member_type` **不接受 `mobile`**——传 `mobile` 会报 `99992402 field validation failed: member_type`。飞书协作者 API 只认这些 member_type：`email` / `openid` / `userid` / `unionid` / `openchat` / `opendepartmentid` / `departmentid` / `groupid` / `appid`。用户给手机号时必须**先调 `POST /contact/v3/users/batch_get_id` 把手机号转成 openid**，再用 `member_type:"openid"` 加协作者。给邮箱则直接 `member_type:"email"`，无需转换。

### 示例：加协作者 curl（邮箱直传）

```bash
curl -s -X POST \
  "https://open.feishu.cn/open-apis/drive/v1/permissions/${APP_TOKEN}/members?need_notification=true&type=bitable" \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "member_type": "email",
    "member_id": "user@example.com",
    "perm": "full_access"
  }'
```

### 示例：手机号转 openid（用户给的是手机号时必跑）

```bash
# 1. 手机号 → openid
curl -s -X POST \
  "https://open.feishu.cn/open-apis/contact/v3/users/batch_get_id?user_id_type=open_id" \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"mobiles":["<MOBILE>"]}'
# 返回 data.user_list[0].user_id = openid（形如 ou_xxxxxxxxxxxxxxxxxxxxxxxx）

# 2. 用 openid 加协作者
curl -s -X POST \
  "https://open.feishu.cn/open-apis/drive/v1/permissions/${APP_TOKEN}/members?need_notification=true&type=bitable" \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d "{
    \"member_type\": \"openid\",
    \"member_id\": \"${OPENID}\",
    \"perm\": \"full_access\"
  }"
```

member_type 优先级：`email` > `openid`（手机号转换后）> `userid`。**不要用 `mobile`，会被拒。** 用户给邮箱时优先 email（让用户收邀请邮件）；给手机号时按上面两步走 openid 路径。

### 错误处理（强制，不能静默跳过）

API 返回非 0 code 时按错误码处理：

| 错误码 | 含义 | 处理 |
|---|---|---|
| `99992402` | `member_type` 字段校验失败（通常是传了 `mobile` 这种不支持的值） | 改用 `email` 或 `openid`；用户给手机号时先调 `batch_get_id` 转 openid 再加 |
| `1254003` | permissions 权限不足（机器人缺 `drive:drive` 之外的什么权限） | 告诉用户检查飞书应用权限第 3 节，重做 |
| `1254043` | 用户不存在（标识填错 / 该用户不在飞书租户内） | 让用户重新给一个有效标识，重试 |
| `1254101` | 标识格式错（手机号没加区号、邮箱拼错） | 让用户重新给，重试 |
| 其他 | 未知错误 | 把飞书原文转告用户，让用户决定 abort 还是继续 |

**任何失败都明确告诉用户**，不要静默继续——否则用户拿到的就是一张自己打不开的表。

### 后续触发不再重问

加协作者只在首次触发建表时执行一次。后续触发时：
- 偏好 sheet 里有 `feishu_user_identifier` + `feishu_id_type` → 直接读出来用
- 用户已在协作者列表里 → 不重复加（API 会返「已存在」错误，忽略即可）
- 用户想加别的飞书账号 → 主动说「我想再加一个协作者」时再加

### 写入失败处理

写入主表时可能遇到两种失败，处理方式分开说：

### 失败 1：飞书 API 报权限错误

表现：调用飞书多维表格 API 返回权限不足、token 失效、表不存在等错误。

处理：
- **不要静默降级到本地文件**（不写 txt / csv / Markdown 兜底），那是数据丢失风险
- 明确告诉用户：「飞书表格写入失败，原因是 {具体错误}。请检查：表格是否共享给本应用？是否还有编辑额度？」
- 等用户处理完反馈后重试同一次写入，不要丢条目

### 失败 2：单条字段缺失

表现：某条新条目缺 `投递链接` 或缺 `公司`（这两个是 `field-contract.md` 3.1 节的必填字段）。

处理：
- 这条**不进入主表**，但也不直接丢——放入**待确认队列**（一个独立的 sheet 或临时列表），告诉用户："X 条因为缺投递链接/公司名进待确认队列，要不要补一下信息？"
- 用户补全后再按追加模式写入主表
- 缺非必填字段（招聘项目、招聘岗位、学历要求、专业要求、是否笔试、行业标签、备注、公告链接等）不触发失败，按字段标注规则填默认值或留空即可

## 旧数据治理：多批次拆行（每次触发 skill 时跑，直到表里无多批次行为止）

**背景**：用户上传的飞书表里"招聘批次"列常以一行多批次形式记录（如"暑期实习、秋招专场""暑期实习,秋招提前批""暑期实习/秋招"），早期搬运时直接照搬到主表 + 子表。这种粒度与 `dedup_judge.md` 环节 B 的 dedup 键 `公司 + 招聘批次`（隐含前提是单值）冲突，且让投递进度变更无法消歧——用户说"投了字节跳动"时 AI 无法定位到具体批次。本节定义一次性治理流程，把存量多批次行全部拆成"一行 = 一个(公司, 批次)"。

### 触发时机

- **每次触发 skill、读完主表 + 子表全部行后立即扫一次**——扫主表「招聘批次」字段是否含常见分隔符（`、` `，` `,` `/` `|` `；` `;` 换行 全角空格 任一），含则进拆行流程
- 首次触发时表里多批次行可能很多（用户上传表规模而定），全部拆完再进 0.5 步推待确认清单
- 后续触发时表里通常已无多批次行（治理过了），扫描结果为空直接跳过本节

### 拆行操作（主表 + 子表双向同步）

对每条「招聘批次」字段含上述常见分隔符的记录：

1. **按常见分隔符拆"招聘批次"为数组** `batches = [b1, b2, ..., bN]`，N ≥ 2（按 `、` `，` `,` `/` `|` `；` `;` 换行 全角空格 任一分隔，命中哪个用哪个）
2. **第一项 b1 update 旧行**：调 `update_record` 改主表该行的「招聘批次」列为 `b1`（单值），其他字段不动
3. **第 2..N 项各 batch_create 一条新行到主表**：字段复制旧行其他列、「招聘批次」填 `bi`、「子表 record_id」列暂空、投递进度按下方"投递进度继承规则"填
4. **每个新主表 record_id 都要双写到对应子表**：
   - 按主表该行「企业性质」列找到对应子表 table_id
   - 调 `batch_create` 写入该子表 22 列（「子表 record_id」列填主表 record_id，「信息更新时间」与主表一致）
   - 拿到子表 record_id 后，update 主表该行的「子表 record_id」列填子表 record_id
5. **旧行对应的子表记录也要同步 update**：旧行的主表 record_id 没变，但「招聘批次」列已改成 b1，对应子表记录按子表 record_id 定位 update「招聘批次」列为 b1
6. **拆完后跑一次 dedup 复判**：拆出的新行可能跟表里已有的单批次行撞公司撞批次（如旧表已有"字节跳动 提前批"一行，又从"字节跳动 暑期实习、秋招提前批"拆出"字节跳动 秋招提前批"）→ 走 `dedup_judge.md` 环节 B LLM 复判流程，high/medium 置信度合并保留更全的一行，low 置信度两行都保留

> 分隔符集只覆盖常见情况（`、` `，` `,` `/` `|` `；` `;` 换行 全角空格）。若用户表用了更怪的分隔符（如 `+` `&&` `·`）导致按上述集合拆出来仍是单行，跳过该行不治理——LLM 复判阶段若发现该行批次字段含异常多值再单独处理，不在本节硬拆。

### 投递进度继承规则

旧行原本的状态决定拆出的所有新行的默认状态：

| 旧行投递进度 | 拆出的新行默认投递进度 | 原因 |
|---|---|---|
| `待确认` | `待确认` | 无歧义，直接继承 |
| `已投递` | `待确认`（强制改回） | 旧行的"已投递"是在粒度不清时标的，无法定位到具体批次——宁可让用户在下一轮 0.5 步清单里重新确认，不要在治理时擅自假设投了哪个批次 |
| `已拒绝` | `待确认`（强制改回） | 同理——用户当时说"不要字节跳动"可能是针对某一批次，不能默认两批都拒绝 |

> 治理的核心目的是消歧，不是保留旧投递进度。强制改回"待确认"是有意为之——下一轮 0.5 步推送会自动把所有刚拆出来的"待确认"行推给用户，用户在新的单批次粒度上重新决定状态，闭环。

### 拆完验证

- 跑「写表后一致性校验」节的全套（校验 1 错位 + 校验 2 孤儿 + 校验 3 数量一致），确认拆行没引入新错位/孤儿/数量漂移
- 重新扫一次主表「招聘批次」字段，确认无 `、` 残留

### 告知用户

拆完后明确告诉用户：

```
旧数据治理完成：本次拆了 X 条多批次记录，产生 Y 条新行
- 这些新行的投递进度都重置为「待确认」
- 下一轮待确认清单会按单批次粒度重新推送给你，请逐条确认投了哪个批次
- 之前在多批次粒度下标的「已投递」/「已拒绝」无法继承到单批次粒度，必须重新确认
```

不要静默拆行——用户看到表里行数变多会困惑，必须主动告知原因 + 闭环到下一轮 0.5 步推送。



每次双写完主表 + 5 子表后，除了字段值 sanity check，**还必须运行下面三类一致性校验**。归类错误、重复和孤儿记录不会自动愈合。

### 校验 1：子表归错位检测（K3/K4/K6）

按 `field-contract.md` 1.6 节的 canonical classify()（见该节末尾伪代码）对**主表每条记录**重算一次 `enterprise_type`，跟该记录当前指向的子表对照：

1. list 主表全部记录 + 5 个子表全部记录，建主表 record_id ↔ 子表 record_id 双向映射（用「子表 record_id」列）
2. 对每个主表记录：
   - 重算 `expected = classify(行业标签, 企业性质)`
   - 找该记录当前指向的子表名 `actual`（按主表「子表 record_id」列所在子表）
   - `expected != actual` → 记入「错位清单」
3. 错位清单非空 → **主动挪**，不要等用户发现再修：
   - 在目标子表 batch_create 一条与源记录同字段值的新记录（「子表 record_id」列填主表 record_id）
   - 拿到新子表 record_id 后，update 主表「子表 record_id」列指向新记录
   - 删除源子表的旧记录（按旧子表 record_id）
   - 这三步顺序不能反——必须先 create→update 主表→delete 旧记录，否则中间态会丢双向映射
4. 挪完后重新跑一次校验 1，确认错位清单为空

> **禁止全删子表后重写**，否则会丢失用户在子表里维护的投递进度和备注。

### 校验 2：子表重复记录检测（K2/K5）

同一个主表 record_id 被多个子表记录指向时，只保留主表当前指向的那条，其他全部删除：

1. 扫 5 个子表全部记录，按每条子表记录的「子表 record_id」列（存的是主表 record_id）建 `main_rid -> [(子表名, 子表 record_id), ...]` 映射
2. 列表中长度 > 1 的，说明主表 record_id 被多个子表记录指向——主表「子表 record_id」列当前指向的那条是「正确」的，其他都是历史 delete 失败留下的孤儿
3. 逐条 delete 不正确的子表记录（按子表名取 table_id + 子表 record_id），失败的累计到 failed 列表
4. 删除后再跑一次校验 2，确认映射里所有 main_rid 列表长度都 = 1

**delete 失败的常见原因**：记录已被用户手动改过 / 子表 record_id 已被飞书内部清理。失败的进入人工复核队列，告诉用户「X 条子表记录 delete 失败，错误码 Y，需手动到飞书界面删除」，不要静默跳过留下脏数据。

### 校验 3：子表合计 == 主表总数（K1）

```
主表记录数 == Σ 5 个子表记录数
```

不等的情况：

- 主表 > 子表合计：有主表记录的「子表 record_id」列指向不存在的子表行（双写时子表 batch_create 失败但主表已写入）→ 按主表数据补写一条到对应子表，回填主表「子表 record_id」
- 主表 < 子表合计：有子表记录的「子表 record_id」列指向不存在的主表行（主表记录被删但子表未同步）→ 删除该子表记录
- 两种情况都修完后，再跑一次校验 3 确认数字一致

### 校验时机

- **每次双写完主表 + 子表后立即跑校验 1 + 2 + 3**——不是拖到下次触发 0.5 步时再跑
- **0.5 步入口 B 同步检测前**也跑一次（用户可能在两次触发间手动改过投递进度/字段，跑一遍能顺便发现错位/孤儿/数量漂移）
- 校验发现异常 → 修完再进 0.5 步推待确认清单，不要带病推送

### 0.5 步投递进度一致性校验的优化（K1）

0.5 步需要兜底修复主表 ↔ 子表投递进度漂移，但不能牺牲正确性。优化只能减少字段和范围，不能改成“只看主表”。

可用优化：
1. list_records 只取必要字段：`record_id`、`子表 record_id`、`投递进度`、`modified_at`、`公司`、`招聘批次`。
2. 如果飞书 API 支持按 `modified_at` 或视图过滤，优先检查最近变更记录；否则定期全量扫。
3. 发现任一端最近变更或值不一致，都按「入口 B 流程」同步。
4. 写表后的双写一致性校验仍保留，用来发现错位、孤儿、数量漂移。

### 失败汇总与用户告知

校验 1/2/3 任何一项修不完时，明确告诉用户：

```
写表后一致性校验未通过，需要你介入：
- 校验 1（子表归错位）：X 条待挪，已挪 Y 条，失败 Z 条（错误码列表）
- 校验 2（子表重复记录）：X 条待删，已删 Y 条，失败 Z 条
- 校验 3（主子表数量一致）：差异 X 条，已修 Y 条，失败 Z 条
请到飞书界面手动处理失败条目，或回复「跳过」继续。
```

不要静默跳过留下脏数据，也不要因为部分失败就 abort 整个 skill——已修好的部分是有价值的，剩下的留给用户决策。
