# 求职企业清单写入与验收

本文件定义 `企业清单` 主表及五张物理分类子表的最小写入协议。目标是增量追加、
稳定映射、保护用户状态；禁止整表重写。

## 1. 表结构

六张企业表使用相同字段和顺序：

<!-- ENTERPRISE_FIELDS:START -->
1. 信息更新时间
2. 投递进度
3. 公司
4. 招聘批次
5. 招聘项目
6. 招聘岗位
7. 公告链接
8. 投递链接
9. 投递截止时间
10. 城市
11. 行业标签
12. 企业性质
13. 子表 record_id
<!-- ENTERPRISE_FIELDS:END -->

字段类型、来源内部键映射和分类枚举以 `field-contract.md` 为准。投递状态只有
`待确认`、`感兴趣`、`已投递`、`已拒绝`。新记录默认 `待确认`。

五张子表固定为：`互联网`、`金融银行`、`外企`、`央国企`、`其他私企`。它们是
真实物理表，不是主表视图。

## 2. 接管与初始化

### 已有 Base

1. 先只读列出表、字段、视图、记录数和 workflow。
2. 不按 Base 名称猜测；以用户提供 URL 解析出的 token 为准。
3. 只报告与 13 字段契约的差异，未获得结构迁移授权前不删除或重排字段。
4. 不覆盖现有 `投递进度` 和 `子表 record_id`。

### 新 Base

按第 1 节建立六张空表；`子表 record_id` 放最后并在普通视图隐藏。建立用户偏好和
信息源登记表时使用各自契约，不能把它们的元数据列加入企业表。

## 3. 增量双写

每条通过筛选与去重的新记录按以下顺序处理：

1. 用规范化投递链接、公告链接和公司+批次+项目建立主表去重索引。
2. 按企业性质选出唯一子表。
3. 写主表 13 字段，其中主表 `子表 record_id` 暂空，取得主表 record ID。
4. 写对应子表 13 字段，子表 `子表 record_id` 填主表 record ID。
5. 将子表 record ID 回填主表 `子表 record_id`。
6. 读回两条记录，确认公司、更新时间、状态和双向 ID 一致。

不生成或维护 `编号`。展示顺序由视图排序决定。来源补全已有记录时只更新空白且属于
来源所有的字段；不得全行覆盖。

## 4. 单边失败修复

- 主表存在、子表缺失：按主表当前 13 字段补建正确子表记录并回填。
- 子表存在、主表技术 ID 为空：核验公司/批次后回填主表。
- 技术 ID 指向不存在记录：补建缺失侧并替换失效 ID。
- 一条企业记录出现多个子表候选：停止自动修复并报告冲突，不删除任何记录。

修复仍采用“先补缺、再读回”的方式；禁止清空子表重建。

## 5. 投递进度闭环

主表与五张子表之间保留 10 条双向 workflow。每条 workflow：

1. 只监听和更新 `投递进度`；
2. 目标定位条件为目标表 `子表 record_id == 触发记录 recordId`；
3. 目标值相同时不写；每次最多更新一条；
4. 排除自动化同步事件以避免循环；
5. 建立后显式启用，并通过主→子、子→主各一次烟测。

Skill 启动时必须检查六张表的状态一致性。修改时间不同取较新值；时间相同且冲突时主表
优先并在摘要提示。

当 Codex 在一次 `job-collection` 运行中将主表状态首次更新为 `已投递` 后，必须在同一次
运行中直接将它 upsert 到独立 `求职进展` Base；以 `企业清单 record_id` 去重。用户在飞书
界面手动改状态时，则由下一次 `job-collection` 运行补偿。不得覆盖求职进展中用户维护的
岗位、JD、首次投递日期或更后的面试阶段。

## 6. 视图

每张企业表至少保留：默认视图、现有批次视图，以及按四种投递状态过滤的状态视图。
新增 `感兴趣` 视图时不得删除用户自定义视图。普通视图默认隐藏 `子表 record_id`。

所有受管 grid 视图排序统一为：

1. `信息更新时间` 降序；
2. `公司` 升序。

删除或重命名旧视图属于结构迁移，只能在已验证备份并得到用户明确批准后执行。

## 7. 写后验收

每次写入或修复后检查：

- 本批主表与子表的 `信息更新时间` 非空且等于来源值；
- 每条主记录只映射一条正确子记录，反向也成立；
- 新记录状态为 `待确认`，已有状态未被来源同步改写；
- 六张表的状态选项包含且只使用四个合法值；
- 10 条双向 workflow 均 enabled；
- 受管视图排序为 `信息更新时间 desc, 公司 asc`；
- `子表 record_id` 是最后字段且在普通视图隐藏。

任一项失败时报告未完成和具体差异，不得把部分成功描述为“同步完成”。

## 8. 结构迁移安全线

字段删除、表删除、批量状态改写和 workflow 替换均不是普通增量同步的一部分。执行前必须：

1. 创建并验证可恢复备份；
2. 保存字段、视图、workflow 和记录数量审计；
3. 向用户列出精确影响并取得明确批准；
4. 迁移后逐项验收，旧 Base 保留为回滚入口。

### 旧数据治理：多批次拆行

一条旧记录含多个真实批次时，先报告拆分计划并保留原状态；获得用户迁移授权后，每个
批次建立独立主子记录。无法判断状态应属于哪个批次时不得复制状态或擅自拆分。

见第 7 节。
