# 腾讯智能表格来源

腾讯 Smartsheet 仅作为浏览器辅助来源。它不是飞书 Base API 来源，不要尝试猜测私有 API、读取 cookies、抓 WebSocket 帧或绕过内容权限。

## 适用条件

- URL 匹配 `https://docs.qq.com/smartsheet/`。
- 用户自己的浏览器已登录，并对文档有查看权限。
- 目标是按日期读取每日新增，或按明确日期边界分批回填历史数据。

任一条件不满足就停止写表。历史回填优先使用腾讯文档导出的 XLSX/CSV；导出/复制不可用时，可在可见行号、日期和链接都能稳定复核的前提下分批视觉扫描。

## Chrome 扩展恢复 SOP

腾讯登录态只允许来自用户自己的 Chrome 会话。开始前完整读取当前安装的 Chrome 控制 Skill，
并按它的连接与清理规则执行：

1. 连接 Chrome extension；首次失败后等待约 2 秒，仅重试一次。
2. 仍失败时运行 Chrome Skill 规定的四项诊断：Chrome 运行状态、浏览器安装、扩展启用状态、
   native host manifest。不得读取 cookie 或手工改 native host。
3. 诊断正常但通道未连接时，在用户已授权的前提下打开 Default Profile 新窗口，再等待约 2 秒
   重试。仍失败则要求从 Codex/ChatGPT 插件界面重装 Chrome 插件，不用 AppleScript、私有 API
   或未登录的内置浏览器替代。
4. 连接成功后立即命名会话并读取浏览器控制文档。优先从 `openTabs()` 认领精确匹配的腾讯标签；
   没有现成标签时在 extension 浏览器中新建标签。内置浏览器能打开标题但要求登录，不算成功。
5. 任务结束前恢复临时 viewport，并按 Chrome Skill 的 `tabs.finalize` 规则释放会话。

## 新用户首次接入

收到腾讯 Smartsheet 链接后，按以下顺序执行，不要先建一个无法继续同步的空来源：

1. 检测浏览器控制能力；没有浏览器能力时，要求用户安装对应浏览器插件，或直接导出 XLSX/CSV。完成标准：Agent 能打开该 URL 并看到表名、视图和权限状态。
2. 在创建目标 Base 时，把腾讯链接写入「信息源登记」：`source_type=tencent_smartsheet`、`source_url=去掉 login_t 等临时参数后的稳定 URL`、`is_active=true`、`is_login_required=true`、`credential_status=browser_session`。`app_token` 和 `table_id` 留空。
3. 打开“文件”菜单并实测导出/复制权限，不要根据“只能查看”四个字猜测。如果“导出为”可用，优先下载 XLSX/CSV并走结构化文件导入；如果导出不可用，再实测表格复制。
4. 导出/复制均不可用时，改走浏览器分批视觉扫描。开始前确定目标日期边界；每批记录首尾行号、首尾更新日期和累计读取数，写入运行检查点。中断后从上次首尾重叠 1-2 行的位置恢复，用公司+日期核对连续性。
5. 视觉扫描不限固定行数，但每行都必须从可见单元格复核字段，并取得可验证的公告或投递入口。可分批写入，`last_sync_time` 只有在扫描到目标日期边界、处理完同日全部行并通过验收后才能更新；中途状态写进 `last_sync_result`，不得伪装完成。
6. 首次回填完成后运行硬筛、去重、双写和完整写后验收。完成标准：源范围已穷尽、待确认行有清单、信息源登记已写回同步基准。

### 首次调用禁止事项

- 不得因为当前 Agent 第一次见到该链接，就跳过浏览器能力和内容权限探测。
- 不得只读取首屏 8-12 行后把 `last_sync_time` 推到最新日期；这会永久漏掉同一日期或更早的未扫描记录。
- 不得把 OCR 结果直接写入生产表。OCR 只能帮助定位候选行，字段和链接必须由可见单元格或点击后的真实 URL 复核。
- 不得因为待扫描行数超过任意固定阈值就直接判定无法完成；应优先导出，失败后使用带行号/日期检查点的分批视觉扫描。只有关键字段不可辨认、行号不连续或链接无法验证时才暂停。
- 不得创建单独的“腾讯文档 skill”；本流程是 `job-collection` 的一个来源分支。

## 增量读取流程

1. 打开用户给出的 URL，确认标题与权限状态。将来源登记为 `source_type=tencent_smartsheet`，保存原始 `source_url`；完成标准是下次 Agent 能只靠「信息源登记」重新定位来源。
2. 展开左侧工作表目录，点击可见的“每日更新”或用户指定工作表；切换后同时验证 URL 的 `tab` 参数或页面公告/表头已经变化。隐藏在 DOM 中但不可见的同名按钮不能作为切换成功证据。定时增量使用该来源自己的 `last_sync_time`，计算 `overlap_start = last_sync_time 所在日期往前 1 天的 00:00:00`；未指定日期时处理所有 `>= overlap_start` 的记录。不要把当前可见的 8-12 行称为全表。
3. 腾讯表复制为空是常见权限行为，不代表没有数据。按屏截图读取可见行；必要时临时扩大 viewport，并横向滚动读取被画布遮住的更新日期、截止日期、公告链接与投递官网链接。每行至少取得：公告标题、更新日期、截止日期、行业、招聘类型、工作地点、公告入口、投递入口。
4. 对每行点击“公告链接”，从新标签页 URL 取得真实公告 URL；不要只保存“点击查看”。投递官网链接可点击时同样取得真实 URL。显示“公告中邮箱投递/扫码投递”时，将该原文写入「投递链接」，并在备注保留说明。
5. 以更新日期分批向下滚动。每屏与上一屏重叠 1-2 行，核对行号连续且没有跳行；记录每批首尾行号、首尾日期、累计读取数和停止位置。完整处理同日全部记录，并看到 `< overlap_start` 的日期后才能停止。行号会随新数据插入而变化，只能校验连续性，不能作为跨日游标。
6. 与飞书来源共用目标主表全局索引：规范化投递 URL 完全相同优先；其次规范化公告 URL；最后公司+招聘批次+招聘项目/公告标题复判。公告 URL 不同且项目/标题也不同的，不得只因公司与批次相同就静默合并。
7. 动态读取「用户偏好」，执行与飞书源表完全相同的 5 条硬筛和批次时间窗。招聘类型含多个真实批次时拆行；“校招/实习”若公告标题只描述实习生招聘，归为一个实习批次，不机械拆成校招和实习两条。
8. 按 `excel-insert.md` 双写主表与对应子表，回填双向 record_id。`信息更新时间` 必须使用腾讯源行的“更新日期”，不得使用 Agent 执行日期。
9. 回读本批主表和子表，确认日期、投递入口、主子映射和投递进度；再确认受管 grid 视图排序为 `信息更新时间 desc, 公司 asc`。不得读取或验收已删除的 `编号`，也不得依赖固定视图数量。异常未修完不得报告成功。
10. 仅在扫描到日期边界、处理完同日全部记录并通过验收后，更新该腾讯来源自己的 `last_sync_time` 和 `last_sync_result`。下次仍重扫两天重叠窗口并去重；浏览器或登录失败时保留旧游标。

## 失败边界

- 页面显示无权限或要求登录：请用户在同一浏览器完成登录/授权后继续。
- 可见行无法辨认关键字段：不写入该行，列入待确认，不猜字段。
- 公告链接点击受浏览器安全策略阻止读取正文：仍可保存已取得的真实 URL 和腾讯行字段；不要绕过限制。
- 只有截图、没有可验证的公告或投递入口：不得写入生产表。

## 验收摘要

每次汇报：读取日期、可见源行数、去重数、硬筛后保留数、主子表新增数、日期缺失数、映射异常数、排序异常数。明确说明这是指定日期增量，不是腾讯文档全量扫描。
