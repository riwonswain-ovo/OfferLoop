# 初始化工作流（skill 内）

> Job Collection 在收集求职偏好、添加自有信息源、首次创建飞书多维表格时应遵循的流程与规则。
>
> **关于本文里出现的赋值记号**：`target_cities = [...]`、`city_filter_mode = "hard"` 这类写法只是给 LLM 看的**字段名 + 取值约定**，不是代码变量——skill 模式下没有数据库、没有变量系统。这些字段实际的落地位置见本文第 6 节"偏好持久化"。

---

## 1. 一次只问一个问题

不要一次抛多个问题。问完一个、用户答完、再问下一个。最多 4-5 个问题后必须能开始首次同步，不要无限收集。

### 交互方式与降级

若当前环境支持交互式选择组件（如 Claude Code 的 `AskUserQuestion`），**优先用选择框让用户点选**；若不支持（Codex、Cursor 等），或 AskUserQuestion 触发后用户没接受选择框而是直接文字回答，立即降级为纯文字列选项。

**降级文字模板**（每次只问一个）：

```
问题：你的毕业年份是？
  1. 2027届
  2. 2026届
  3. 2028届
  4. 其他（请直接回复）
请回复编号或直接打字。
```

用户答完直接进下一题，不要回退到 AskUserQuestion 重试。

### 1.1 先判断是否已有目标 Base

开始收集新用户偏好前，先问用户是否已有正在使用的求职 Base。用户提供现有 Base URL 时：

1. 解析并只读检查真实 Base，不按文件标题判断版本；历史名称和用户自定义名称均兼容。
2. 发现 `企业清单` 或 `用户偏好` 任一表时，立即切换到已有 Base 接管模式，不再执行新建 Base 流程。
3. 读取现有用户偏好，仅询问空值或本次同步必需但不存在的字段。
4. 现有招聘记录、编号、投递进度、备注和 record_id 映射一律原地保留。
5. 结构升级只能新增缺失字段、表、视图或 workflow；禁止删除旧表后重建，禁止全量覆盖记录。
6. 用户明确表示没有现有 Base 时，才继续下面的首次偏好问题和新建流程。

## 2. 用户偏好问题

### 2.1 毕业年份（必问）

询问用户是哪一届毕业生，如「2027届」「2026届」。必填，用于源表届次硬筛和批次时间窗判断。

届别与招聘批次的换算见 `SKILL.md` 第三步"届别规则"：
- 搜暑期实习、提前批、秋招 → 优先用**下一届**届别
- 搜春招、补录 → 优先用**当前届**届别
- 用户明确给届别 → 以用户为准

存到字段 `graduation_year`。

### 2.2 目标城市（必问）

接受如下输入：

```text
北京 上海
上海,杭州
全国
```

存储到 `target_cities`。规则：

- 用户答「全国」→ `target_cities = ["全国"]`、`city_filter_mode = "soft"`（避免因条件过宽把所有企业过滤掉）
- 用户答具体城市 → `target_cities = ["北京", "上海", ...]`、`city_filter_mode = "hard"`

> 赋值仅为本轮会话内的字段约定，落地方式见第 6 节。

### 2.3 目标行业（必问）

让用户多选。推荐 14 项取值（13 个行业 ID + 1 个兜底「其他」），与 `references/field-contract.md` 1.3 节 `industry_module` 一致：

| ID | 名称 |
|---|---|
| `internet` | 互联网与科技 |
| `finance` | 金融 |
| `fmcg` | 快消与零售 |
| `manufacturing` | 制造业 |
| `newenergy_auto` | 新能源与汽车 |
| `healthcare` | 医疗与健康 |
| `education` | 教育 |
| `realestate` | 房地产与建筑 |
| `culture_media` | 文化传媒与娱乐 |
| `energy_chem` | 能源与化工 |
| `crossborder` | 出海与跨境 |
| `marketing_consulting` | 广告营销与咨询 |
| `central_soe` | 央国企 |
| `other` | 其他（兜底，命中不到上面的就选这个；也允许用户自行填写其他关键词，不强匹配） |

**AskUserQuestion 选项文字必须精简**，确保 14 项在一屏可见；若 UI 工具列不下 14 项，分组显示或降级为文字列选项。**同时加「自定义填写」入口**——用户输入命中不到 13 个 ID 的关键词时不强匹配，直接把该关键词作为额外约束落到偏好画像表的 `selected_industries` 字段（追加，不替换 ID 列表）。

用户可选一个或多个。`selected_industries` 字段以 ID 列表形式存储，自定义关键词追加在数组末尾。

> 不再问"目标企业类型"——这层分类与目标行业重复，且不参与源表硬筛。企业性质由同步时自动分类。

### 2.4 目标公司（可选）

询问用户是否有重点目标公司。可接受「字节跳动 腾讯」「阿里巴巴,美团」「无」。

- 答「无」→ `target_companies = []`
- **只作加分项，不作硬筛选**——非目标公司的企业只要其他条件匹配也可进入候选

### 2.5 排除项（可选）

收完目标行业后问一句："有没有不想投的公司 / 行业？"不强制，用户说"没有"就跳过。两种排除项：

- **排除公司** `excluded_companies`：明确不想投的特定公司。例：用户说"不投字节、不投拼多多"→ 把这两个公司名写入。匹配时看公司名是否命中
- **排除行业** `excluded_industries`：明确不想进的整个行业。例：用户说"不进金融、不进央国企"→ 转 ID 写入 `["finance", "central_soe"]`。取值同 `selected_industries`，见 2.4 节 13 个 ID。匹配时看企业的 `industry_module` 字段是否命中

排除是**强过滤**——两条任一命中，默认不推送，除非用户本轮明确要求看某条被排除的企业。

> 不收集"排除招聘批次"。本 skill 搜的是企业，一个企业的招聘批次是事实信息（开了就是开了），按批次过滤会把整条企业线索误伤掉。批次层面取舍交给用户看清单时自行判断。

与目标行业冲突时（用户在 `selected_industries` 和 `excluded_industries` 同时选了同一个 ID）：当场提醒用户矛盾，问是想保留为目标还是排除，二选一。不要默默合并或默默清空。

### 2.6 飞书登录标识（首次必问，后续触发不重问）

**只在首次触发时问一次**——后续触发时该标识已存在偏好 sheet 里，直接读出来用。

询问用户：「为了让建好的飞书表你能直接在浏览器打开（机器人建的表默认 owner 是机器人自己，你本人没权限），我需要把你的飞书账号加成表协作者。你的飞书是用什么登录的？给我对应的那一个：
1. 手机号（直接给号码）
2. 邮箱（直接给邮箱地址）
3. 微信登录 → 给微信绑定的手机号或邮箱

**这个标识会跟飞书 App 凭证一起存在偏好 sheet 里，不进 git、不进 commit；如果不给标识，机器人建完表你打不开，得自己重新让飞书管理员帮你转 ownership。**」

存储到 `feishu_user_identifier`（用户给的字符串）+ `feishu_id_type`（`mobile` / `email`）。

> **K8 卡点（重要）**：`feishu_id_type` 存 `mobile` 没问题，但**加协作者时不能直接把 `mobile` 传给飞书**——协作者 API 的 `member_type` 不接受 `mobile`，会报 `99992402`。4.1 节第 5 步加协作者前，若 `feishu_id_type=mobile` 必须先调 `POST /contact/v3/users/batch_get_id` 把手机号转成 `openid`，再用 `member_type:"openid"` 调加协作者 API。`feishu_id_type=email` 时直接 `member_type:"email"` 无需转换。

> 这个字段跟 2.1-2.5 的求职偏好不同——它是「让 Agent 能给你的表加权限」的操作凭证类信息，但仍属于用户画像范畴，存偏好 sheet 跟求职偏好一起持久化。

## 3. 自定义信息来源

询问用户是否要添加自有信息源：

- 答「无」→ 暂停初始化；本 skill 至少需要一个飞书 Base 或腾讯 Smartsheet 来源
- 用户发 URL → 按下方流程登记、补齐必要信息，再问"还有更多信息源吗"

### 3.1 分类规则

按 URL 域名判类型：

| URL 特征 | 类型 | 处理文档 |
|---|---|---|
| 含 `feishu.cn/base/`、`larksuite.com/base/` | 飞书多维表格 | `references/personal-excel-source.md` 第二节 |
| 含 `docs.qq.com/smartsheet/` | 腾讯智能表格 | `references/tencent-smartsheet-source.md` |
| 其他 URL | 不支持 | 请用户改为提供飞书 Base 或腾讯 Smartsheet 链接 |

**飞书 / 腾讯文档表格的读取、字段映射、合并规则全部集中在 `references/personal-excel-source.md`**，本节只负责登记。登记后告诉用户："表格内容会作为最高优先级信息源，结果主体以此为准。"

### 3.2 是否需要登录

| 类型 | 默认 |
|---|---|
| 飞书多维表格 | `is_login_required = false`（要求用户将 AI 应用加入协作者） |
| 腾讯文档表格 / 智能表 | `is_login_required = true`（走浏览器登录态读取，详见 `references/tencent-smartsheet-source.md`） |

### 3.3 登记流程

用户发 URL → AI 自行判断 `source_type` 和 `is_login_required`，**只问缺失的必要信息**，不要把已能自动判断的再问一遍。登记字段以「信息源登记」sheet（见 6.1 节 Sheet 3）为准：

- `source_id`（用户自定义源用 `SRC_CUSTOM_YYYYMMDDHHMMSS`）
- `source_name`
- `source_url`
- `source_type`（仅 `feishu_bitable` / `tencent_smartsheet`）
- `app_token` / `table_id`（飞书源表必填，从 URL 解析——解析规则见 `references/personal-excel-source.md`「飞书源表 URL 解析」节）
- `is_login_required`
- `is_active = true`
- `credential_status`（`not_required` / `pending` / `valid`）
- `added_at`（登记时间戳）

**登记即持久化**——不要只存在对话上下文里。登记流程：调飞书 `POST /bitable/v1/apps/{app_token}/tables/{信息源登记 sheet table_id}/records` 把字段写入新行。下次触发时 0.6 步靠这张 sheet 定位所有源表，不依赖 /tmp 或对话上下文。

登记完再问"还有更多信息源吗"。用户后续上传第二个、第三个飞书源表时同样按本流程 append 新行到信息源登记 sheet——**支持任意数量源表长期记忆**，不再退化到单源表 schema。

### 3.4 腾讯浏览器登录引导

如果来源是腾讯 Smartsheet：

1. 要求用户在自己的 Chrome 中登录腾讯文档，并确认链接可见。
2. 探测是否具备控制用户现有 Chrome 的能力；不接受用户把 Cookie/JWT 粘贴进对话。
3. 浏览器能力可用时按 `references/tencent-smartsheet-source.md` 执行。
4. 浏览器能力不可用时，优先请用户从页面导出或复制结构化内容；两者都不可用则暂停该来源，不推进游标。

合规边界：用户访问自己有权查看的账号数据，不涉及破解反爬、过验证码。

## 4. 飞书多维表格初始化

> **飞书未对接时跳过本节**——直接走第 6.3 节"未持久化偏好 sheet"路径，第 7 节反馈里也明确标注"未持久化"。

仅当用户明确表示没有现有目标 Base 时，才按 `references/excel-insert.md` 定义的 22 列在用户飞书工作空间里创建新表。不得因为当前 Agent 第一次加载本 Skill，就忽略用户已存在的 Base。

例外：用户主动说"我之前已经有一个类似招聘信息表"——让用户发链接，按 `references/personal-excel-source.md` 处理；处理失败（访问被拒、明显不是招聘信息表）再退回创建新表。

新表创建好后，记下表格的飞书 URL，等第 7 节末尾反馈给用户。

### 4.1 顺序：先收集必要信息，再建 Base + 8 个 sheet 并写入

**飞书对接成功后立即执行：**

1. 在用户飞书工作空间创建多维表格文件
2. 在该文件里建 8 个 sheet，按以下顺序：
   - **先建「用户偏好」sheet**（按 6.1 节 Sheet 2 结构，10 列求职画像字段）
   - **再建「企业清单」主表 sheet**（按 `excel-insert.md` 22 列结构）
   - **再建 5 个子表**（互联网 / 金融银行 / 外企 / 央国企 / 其他私企，每个子表 22 列结构与主表一致，建表流程见 `excel-insert.md`「5 个子表建表」节）
   - **最后建「信息源登记」sheet**（按 6.1 节 Sheet 3 结构，12 列字段——`source_id` / `source_name` / `source_type` / `source_url` / `app_token` / `table_id` / `is_login_required` / `is_active` / `credential_status` / `last_sync_time` / `last_sync_result` / `added_at`，类型全用 Text 除 `last_sync_time` / `added_at` 用 DateTime）
3. 画像 sheet 按第 6.1 节 Sheet 2 结构建好列，写入本轮已收集到的偏好（10 列求职画像字段）
4. 主表 sheet + 5 子表都建好先空着——首次读取来源后按 `excel-insert.md`「追加模式（双写主表 + 子表）」流程写入
5. 信息源登记 sheet 建好先空着——3.3 节登记用户上传的源表时才 append 行
6. 8 个 sheet 都建好后**立即加用户为协作者**（不要拖到首次同步完成后）：
   - 调 `POST /open-apis/drive/v1/permissions/{app_token}/members?need_notification=true&type=bitable`
   - `member_type` 取值看 2.6 节收集的 `feishu_id_type`：
     - `feishu_id_type=email` → `member_type:"email"`，`member_id` 直接用 `feishu_user_identifier`
     - `feishu_id_type=mobile` → **必须先转 openid**：调 `POST /contact/v3/users/batch_get_id` body `{"mobiles":["<手机号>"]}` → 取返回 `data.user_list[0].user_id` → 再用 `member_type:"openid"` + `member_id=<openid>` 调加协作者 API。**不要直接传 `member_type:"mobile"`，会报 `99992402`**
   - `perm` = `full_access`（用户可编辑，机器人 + 用户都有操作权限）
   - **API 失败必须 abort 整个 skill**——不能静默继续。明确告诉用户：「加协作者失败，原因 {飞书返回的错误码 + msg}。你暂时无法在浏览器打开这张表。可选：①重新给一个有效飞书标识让我重试 ②我先继续搜，搜完再统一加」。用户选 ②的话把 abort 标识记进对话上下文，第九步末再补一次加协作者。
   - 成功后立即告诉用户：「已加你为协作者，邀请邮件已发到 {标识}，飞书云文档 → 我的空间也能搜『Job Collection 求职企业清单』找到它。」
7. 加协作者成功后再告诉用户画像 sheet 的飞书 URL，然后开始首次来源同步
8. 5 个子表的 table_id + 信息源登记 sheet 的 table_id 记录在对话上下文里，下次触发时按 base app_token 重新 list tables 拿回

**不要把建子表 / 信息源登记 sheet 拖到首次读取后**——子表建好后才能双写，信息源登记 sheet 建好后来源才能持久化。

**不要把画像 sheet 拖到最后跟同步结果一起建**——偏好只在对话上下文里会随会话结束丢失。

如果本轮偏好还没收集完就想先建表，**等偏好收集完再建**——画像 sheet 一次性写入完整偏好，不要建空 sheet 占位。

### 4.2 主表 / 子表空表时的占位行处理

跑通过程中曾出现主表数据从第 10 行开始而非第 1 行——前 9 行是 `batch_create` API 留下的占位行。建表完成后立刻做这件事（主表 + 5 个子表都要做）：

1. 建表后立即调飞书 API 读取该表全部行
2. 发现前若干行是空行（关键字段全空）→ 调用删除行 API 把这些空行清掉
3. 再写入首批 5 条样本前确认表头在第 1 行

如果用 `insert_records` 指定从第 1 行插入可以避免占位行问题——优先用这个 API。具体 API 名以飞书当前文档为准。

## 5. V0.1 限制

V0.1 阶段不向用户询问：字段映射、CSS 选择器、正则解析规则、自定义解析逻辑。直接用 LLM 抽取 + 字段契约规则。如果遇到抽取失败的具体页面，临时让用户粘贴文本兜底，不要让用户配解析规则。

## 6. 偏好持久化（飞书用户偏好 sheet）

skill 模式下没有数据库，用户偏好要跨会话保留，靠飞书多维表格里的**用户偏好 sheet**持久化。

**绝对顺序（首轮必须遵守）：飞书对接成功 → 收集偏好、飞书用户标识和至少一个支持的信息源链接 → 创建 Base + 8 个 sheet → 写入用户偏好与信息源登记 → 加用户为协作者 → 首次同步。**

### 6.1 sheet 结构

在用户的飞书多维表格里开以下三个 sheet：

**Sheet 1：企业主表（默认名「企业清单」）**
- 列定义见 `references/excel-insert.md`（22 列，含『企业性质』和『子表 record_id』两列）
- 每条企业线索一行
- 主表是单一事实源，5 个子表是分类视图（互联网 / 金融银行 / 外企 / 央国企 / 其他私企），每条线索双写主表 + 对应子表，详见 `excel-insert.md`「追加模式（双写主表 + 子表）」节
- 主表和 5 个子表都必须创建 `已投递` 过滤视图；列顺序继承各自默认视图，排序为 `信息更新时间 desc, 编号 desc`
- 必须创建并启用主表 ↔ 5 个子表的 10 条投递进度双向 workflow。用户在任一主表、子表或其视图手动改状态后，底层配对记录同步，相关 `已投递` 视图自动进入或移出该记录

**Sheet 2：用户偏好（单行记录）**
- 单行记录，10 列求职画像字段（源表元数据已移到 Sheet 3，不再放画像 sheet）：

| 列名 | 说明 |
|---|---|
| `graduation_year` | 毕业年份，如 `2027届` |
| `target_cities` | 目标城市列表，逗号分隔，或填 `全国` |
| `city_filter_mode` | `hard` 或 `soft` |
| `target_companies` | 目标公司列表，逗号分隔；没有留空 |
| `selected_industries` | 行业方向列表，逗号分隔（取值见 field-contract.md 1.3 节） |
| `excluded_industries` | 排除行业列表，逗号分隔（取值同上） |
| `excluded_companies` | 排除公司列表 |
| `feishu_user_identifier` | 飞书登录标识（手机号或邮箱），用于建表后加协作者 |
| `feishu_id_type` | 标识类型：`mobile` 或 `email`。注意：`mobile` 不能直接传给协作者 API，4.1 节第 5 步会先转 `openid` 再加 |
| `updated_at` | 最近一次更新时间（ISO 日期时间） |

> **向后兼容历史数据**：本规则上线前已建好的偏好 sheet 可能有 14 列（含 `source_table_app_token` / `source_table_table_id` / `source_table_name` / `last_sync_time`）——不要删这 4 列。若 `信息源登记` 为空，允许仅读一次这 4 列迁移出一条信息源登记记录；迁移后 0.6 步只读 `信息源登记`，不再把这些旧字段作为兜底真源，也不再写入新值。

**Sheet 3：信息源登记（多行记录，支持任意数量源表）**
- 多行记录，每行一个源表，12 列：

| 列名 | 类型 | 说明 |
|---|---|---|
| `source_id` | Text | `SRC_CUSTOM_YYYYMMDDHHMMSS`，唯一标识 |
| `source_name` | Text | 用户可读名，如「第三方招聘汇总表」 |
| `source_type` | Text | `feishu_bitable` / `tencent_smartsheet` |
| `source_url` | Text | 原始 URL |
| `app_token` | Text | 从 URL 解析，飞书源表必填（解析规则见 `personal-excel-source.md`「飞书源表 URL 解析」节） |
| `table_id` | Text | 从 URL 解析，飞书源表必填 |
| `is_login_required` | Text | `true` / `false` |
| `is_active` | Text | `true` / `false`（用户暂停某个源表时改 false，0.6 步跳过） |
| `credential_status` | Text | `not_required` / `pending` / `valid` |
| `last_sync_time` | DateTime | 该 source_id 独立的成功扫描高水位。每日任务从其日期前一天 00:00 重扫；失败时不推进 |
| `last_sync_result` | Text | 扫描窗口、候选/重复/新增/补全/失败数、游标前后值或失败原因 |
| `added_at` | DateTime | 登记时间戳 |

> **0.6 步靠 Sheet 3 定位所有源表**——不要把源表元数据只存 /tmp 或对话上下文。/tmp 跨会话可能被清，对话上下文压缩时会丢；只有落到 Sheet 3 才能保证下次触发 0.6 步时拿得到、且支持多源表循环同步。
> 每行游标彼此独立。飞书来源成功、腾讯浏览器失败时，只推进飞书来源那一行；禁止用一次任务的全局时间覆盖两行。
>
> **"待确认队列"不持久化**——`excel-insert.md` 失败 2 里提到的"待确认队列"只活在当前会话的对话上下文里（缺 `投递链接` / `公司` 的条目），不另开飞书 sheet 持久化，会话结束未补全的条目丢弃。下次触发如果用户记得要补，AI 重新搜一次即可。

### 6.2 读写规则

- **首次初始化**：表格里还没有 `用户偏好` sheet → 创建并写入第一次收集到的偏好
- **已有 sheet**：下次触发 skill 时，先读 sheet 拿到历史偏好，**只在用户本轮明确调整某个字段时才更新对应列**（不要每次都全部重问）
- **追加列**：新增偏好字段时，往表里加新列，不要破坏旧列

### 6.3 飞书表创建失败或用户拒绝创建时

按第 4 节默认创建新表后，仍可能遇到创建失败 / 用户拒绝授权 / 用户暂时不想建表的情况。此时偏好**只能在当前会话内用对话上下文维持，会话结束就丢**，下次触发要重新收集。

处理顺序：

1. 先尝试创建 → 失败（飞书 API 不通、未授权、超额度）→ 告诉用户具体原因
2. 用户明确说"先不建表" → 跳过创建，本轮偏好进对话上下文
3. **第 7 节的初始化反馈照样给**，但备注里写明"未持久化偏好 sheet，下次触发需重新收集偏好"，并提示用户：「建议下次直接让我建好飞书表，偏好就能跨会话保留，不用重新填一遍。」

## 7. 初始化完成后的反馈（必发）

走完偏好收集、信息源登记、飞书表初始化后，**必须给用户一条明确反馈**，至少包含：

- ✅ **飞书多维表格的位置**：
  - 创建新表 → 给出表格的飞书 URL（可点击）
  - 用已有表 → 确认已使用用户提供的链接
  - 创建失败 / 用户拒绝 → 明确说"未持久化偏好 sheet"，并给出后续怎么办
- ✅ **本轮收集到的偏好摘要**（一段话，便于用户核对）：毕业年份 / 目标城市 / 企业类型 / 行业方向 / 排除项 / 目标公司
- ✅ **已登记的信息源**：逐条列出 `source_name` + 类型 + 是否需登录；需登录的标"待你在 Chrome 里登录好之后告诉我"
- ✅ **下一步**："现在开始按以上偏好搜企业，搜到的结果会写入上面的飞书表。"

模板示例：

```
📋 初始化完成

▸ 飞书表：{表格名}（{飞书 URL}）
  └ 偏好 sheet 已创建并写入本次偏好

▸ 本轮偏好：
  - 毕业年份：2027届
  - 目标城市：北京、上海、杭州
  - 行业方向：互联网与科技、央国企
  - 排除：金融行业、字节跳动
  - 目标公司：无

▸ 信息源：
  1. 已有飞书招聘表（{URL}）—— 不需登录
  2. 腾讯招聘汇总表 —— 浏览器会话已登记

下一步：现在开始从已登记来源同步，结果会写入上面的飞书表。
```

## 8. 一句话总结

初始化工作流的目标是：在对话内收集用户求职偏好、登记自有信息源、确保飞书多维表格可用。流程尽量简单、分步、低负担，**走完必须给反馈**，让用户知道表在哪、偏好是什么、Agent 接下来要做什么。
