# OfferLoop 字段与流程验收清单

本清单用于用户确认后的只读检查和合成端到端演练。真实飞书资源缺字段时只报告差异；不要
自动创建或重排字段。

## 招聘信息 → 求职进展

招聘信息事实源是 `求职企业清单`。主表和企业性质子表的业务字段顺序应为：

1. 信息更新时间
2. 投递进度
3. 公司
4. 招聘批次
5. 招聘项目
6. 招聘岗位
7. 公告链接
8. 投递链接
9. 投递截止时间
10. 城市
11. 行业标签
12. 企业性质
13. 子表 record_id（技术字段，放最后）

投递进度只能使用：`待确认`、`感兴趣`、`已投递`、`已拒绝`。求职进展第一次收到“已投递”
时才创建记录；公司可自动带入，但投递岗位、岗位 JD 保持空白，不能从招聘岗位字段复制长文本。

重复同步不得覆盖用户手填的投递岗位、岗位 JD、首次投递日期或更高面试阶段。
`企业清单 record_id` 是可重复的父级关联键；`投递记录 ID` 是每次岗位投递的唯一键。
同一企业下两个不同岗位必须保留为两条进展，分别拥有不同的 `投递记录 ID`，同步时不得报重、
合并或删除。

## 笔面试中心 → 求职进展

`笔面试中心` 的物理表为：`全部安排`、`笔试`、`群面`、`一面`、`二面`、`三面`、`HR面`。
同公司不同岗位及同岗位的多轮面试都应保存为独立事件。

各表共同字段顺序：

1. 安排名称（公式主字段，不含时间）
2. 环节
3. 公司
4. 业务线
5. 岗位
6. 关联求职记录
7. 开始时间
8. 结束时间
9. 截止时间
10. 笔试类型
11. 笔试子类型
12. 平台
13. 链接
14. 注意事项
15. 面试准备文档
16. 面试复盘文档
17. 完成状态
18. 求职记录ID
19. 来源邮件ID
20. 日历状态
21. 已建日程ID
22. 子表 record_id

笔试记录的“面试准备文档”和“面试复盘文档”必须为空。未明确轮次的面试只进入主表，
环节为“面试（轮次待确认）”，且不推进求职进展。

阶段只能单调推进：

```text
已投递 → 笔试 → 群面 → 一面 → 二面 → 三面 → HR面
```

`Offer` 与 `已结束` 仅由用户设置。任何迟到、重复或较早轮次邮件都不能降低阶段。

## 端到端最小验收

1. 合成一条招聘信息，并确认企业清单只写入字段契约中的值。
2. 将其投递进度改为“已投递”，确认求职进展创建/更新而不填入投递岗位和岗位 JD，并生成
   唯一 `投递记录 ID`。
3. 合成一封明确轮次的面试通知，确认生成一个独立笔面试事件、关联正确申请并单调推进阶段。
4. 在同一企业父级下保留两条不同岗位申请，再重放“已投递”事件，确认两条都保留、来源链接
   都刷新、岗位/JD/阶段不被覆盖。
5. 合成同公司不同岗位的一面通知，确认不会合并到前一条申请。
6. 合成无明确轮次面试，确认它只停留在主表且不推进阶段。
7. 在每个写入或日历创建动作前保留人工确认；验收默认使用 dry-run 或本地合成对象，不碰真实数据。
