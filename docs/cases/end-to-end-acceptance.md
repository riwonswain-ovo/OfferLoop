# OfferLoop 合成端到端验收用例

此用例用于在发布到 GitHub 前跑通整体流程。默认只使用本地合成对象、dry-run 和只读核验；
不要把它直接应用到用户真实飞书记录、日历或邮箱。

最近一次发布前的脱敏本地执行结果见
[release-acceptance-2026-07-20.md](release-acceptance-2026-07-20.md)。

## 前置条件

- `offerloop-setup` 对 `full` 的离线预检已完成；任何 `blocked` 或 `needs_action` 项都已明确处理。
- 三张 Base、知识库首页、工作台 URL 和可选 `progress_sync` 定位器均已登记。
- 真实 IMAP、飞书写入与日历创建仍需在各自步骤获得单独确认。

## 用例 A：招聘信息与投递进展

输入一条合成招聘信息：公司为“示例科技”，岗位为“AI 产品经理”，投递进度为“待确认”。

预期：

1. 企业清单只产生字段契约规定的信息；`子表 record_id` 保留在最后。
2. 将投递进度改为“已投递”后，求职进展以企业清单 record ID 创建或更新一条记录。
3. 新记录的公司可带入，但投递岗位和岗位 JD 都为空。
4. 对同一企业清单 record 再次同步时，不覆盖用户填写的投递岗位、岗位 JD、首次投递日期和更高阶段。

## 用例 B：明确轮次的面试通知

输入一封合成邮件：示例科技、AI 产品经理、一面、固定时间。先执行事件抽取和关联 dry-run。

预期：

1. 候选结果进入“全部安排”和“一面”子表的逻辑路径。
2. 关联到用例 A 的唯一求职记录。
3. 仅在用户确认 Base 写入后才创建事件；仅在用户第二次确认日历方案后才创建日程。
4. 求职进展从“已投递”单调推进到“一面”。
5. 面试准备文档、面试复盘文档字段保留为空，供后续 Skill 使用。

## 用例 C：同公司多岗位与轮次待确认

1. 为同一公司再创建一条不同岗位的合成投递记录，并输入该岗位的一面通知。
   - 预期：产生独立事件并关联正确岗位，不合并到第一条申请。
2. 输入一封仅写“技术面”的合成邮件，未写明一面/二面。
   - 预期：只进入“全部安排”，环节为“面试（轮次待确认）”，不创建阶段子表记录，不推进求职进展。

## 用例 D：重复、改期和工作台

1. 重新输入相同来源邮件 ID。
   - 预期：不重复创建事件或日程。
2. 输入引用原邮件 ID 的改期通知。
   - 预期：更新原事件，不创建第二个同一事件。
3. 只读打开工作台 URL 和知识库首页。
   - 预期：入口可访问；业务数据展示、视图切换和未来 7 天模块均不需要复制真实数据到新系统。

## 通过标准与记录

每个用例记录：执行模式（合成/dry-run/只读/已确认写入）、结果、失败原因和待补偿项。通过
不等于已向真实数据写入；任何真实写入、日历创建或邮件读取都必须有单独的用户确认记录。
